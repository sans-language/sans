# v0.7.0 — Foundations & Bug Fixes

**Date:** 2026-03-20
**Status:** Draft
**Goal:** Make the compiler error experience usable and fix critical correctness bugs
**Branch:** `feature/0.7.0-foundations`

## Overview

v0.7.0 is the foundation for all subsequent 0.7.x work. It threads source locations through the compiler pipeline, replaces the crash-on-first-error model with structured multi-error diagnostics, fixes four critical bugs, and establishes Linux CI.

## 1. Source Location Propagation

### Problem
The lexer captures line:col in every token (32-byte struct with kind, value, line, col) but:
- No accessor functions exist (`tok_line`, `tok_col`) — data is stored and lost
- AST nodes contain zero location fields — typeck/IR/codegen cannot report where errors occur
- All compiler errors are locationless strings

### Design

**1a. Token accessors** (lexer.sans or parser.sans):
```
tok_line(t:I) I = load64(t + 16)
tok_col(t:I) I = load64(t + 24)
```

**1b. Uniform "last two slots" convention for AST nodes:**

Every expression and statement node gets two new fields appended at the end: `line:I` and `col:I`. To avoid per-node-type offset lookups, store the total node size at offset 0 alongside the tag, OR use a simpler approach: store line:col at **fixed negative offsets from node end**.

Recommended approach — **size-aware accessor**:
- Each `make_expr_*` / `make_stmt_*` function already knows its node size
- Add `line` and `col` as the last two i64 fields in every node
- Add a `node_size` field at offset 8 of every node (shift existing fields by +8)
- Layout: `[tag:I, size:I, ...fields..., line:I, col:I]`
- Accessors:
  ```
  node_line(n:I) I = load64(n + load64(n + 8) - 16)
  node_col(n:I) I = load64(n + load64(n + 8) - 8)
  ```

This means ONE accessor pair works for all 20+ expression types and 10+ statement types, regardless of their field layout.

**1c. Parser changes:**
- Every `p_parse_expr_*` and `p_parse_stmt_*` function captures `line = tok_line(p_peek(p))` and `col = tok_col(p_peek(p))` at entry
- Passes them to the `make_*` constructor which stores them in the last two slots

**1d. Downstream usage:**
- Typeck: `node_line(expr)` / `node_col(expr)` in error messages
- IR: carry source location in IR instruction metadata (see section 2)
- Codegen: use for future DWARF debug info (v0.9), not used in 0.7.0

### Files Changed
- `compiler/parser.sans` — add tok_line/tok_col accessors, expand all make_expr_*/make_stmt_* constructors, update all parse functions
- `compiler/constants.sans` — no changes needed (node tags unchanged)
- `compiler/typeck.sans` — use node_line/node_col in error paths
- `compiler/ir.sans` — propagate location into IR instructions (optional in 0.7.0)
- `compiler/codegen.sans` — update all AST field offset reads (+8 shift for new size field)

### Risk
This touches every AST constructor and every parse function — high line count but low complexity per change. The "last two slots" pattern means existing field offsets shift by +8 (for the new size field) in every node type. This is the most invasive change in 0.7.0 and should land first.

## 2. Structured Diagnostics

### Problem
The compiler uses `p(msg); exit(1)` everywhere — one error kills compilation, no file:line:col, no source context, no warnings.

### Design

**2a. Diagnostic struct** (40 bytes):
```
[severity:I, line:I, col:I, message:I, suggestion:I]
```
- Severity: `DIAG_ERROR = 0`, `DIAG_WARN = 1`, `DIAG_INFO = 2`
- message: string pointer
- suggestion: string pointer (0 = none)

**2b. Diagnostic list:**
A global array (or context-threaded array) that collects diagnostics instead of exiting.

**2c. Error recovery strategy:**
- **Parser errors: remain fatal in 0.7.0.** Parser error recovery (skip to next statement boundary) is complex and deferred. One parse error = one diagnostic + exit.
- **Typeck errors: multi-error.** Replace `tc_error(msg)` calls with `tc_diag(severity, node, msg)` which extracts line:col from the AST node and appends to the diagnostic list. Typeck continues checking subsequent statements.
- **IR/codegen errors: remain fatal.** These indicate compiler bugs, not user errors.

**2d. Rendering (main.sans):**
After typeck completes, if any DIAG_ERROR exists, print all diagnostics and exit(1). Format:
```
file.sans:12:5: error: type mismatch in 'x': expected I, got S
   12 |   x = "hello"
      |       ^^^^^^^
file.sans:18:3: warning: unused variable 'y'
   18 |   y = 42
      |   ^
```

Source line display requires keeping the source text accessible during typeck. Currently, each module's source is read in `main.sans` (`file_read`), passed to the lexer, but the raw string is not retained after lexing. Fix: store each module's source text pointer in the typeck context (add a `source:I` field to the typeck context struct, set during `build_step3` when type-checking each module). The rendering function in `main.sans` can then index into the source string by scanning for the Nth newline to extract the error line.

**2e. Initial warnings (typeck):**
- Unused variables (declared but never referenced in scope)
- Unreachable code after unconditional return
- Shadowed variable names deferred to a later release (lower priority than the above two)

**2f. Exit behavior:**
- Any DIAG_ERROR → exit(1) after printing all diagnostics
- Warnings alone → compilation continues, warnings printed to stderr

### Files Changed
- `compiler/typeck.sans` — replace ~50 `tc_error()` calls with `tc_diag()`, add warning detection
- `compiler/main.sans` — diagnostic rendering, source text retention
- `compiler/constants.sans` — add DIAG_* severity constants

### Dependency
Requires source location propagation (section 1) to be complete first.

## 3. Critical Bug Fixes

### 3a. `random()` Returns `max/2`

**File:** `runtime/math.sans`, lines 13-15
**Current:** `sans_random(max:I) I { if max <= 0 { 0 } else { max / 2 } }`
**Fix:** Call libc `rand()` and return `rand() % max`. The runtime already seeds with `srand(time(0))`. Declare `rand` as extern if not already declared.
**Test:** Call `random(1000000)` 10 times, verify at least 2 distinct values are produced (avoids flaky single-comparison tests).

### 3b. JSON Float Truncation

**File:** `runtime/json.sans`, lines 1069-1163
**Problem:** `sans_json_p_make_number` calls `sans_json_p_atoi` which stops at `.`, so `3.14` → `3`.

**Fix:**
1. In `sans_json_p_make_number(px, start, end)`: scan substring [start, end) for `.` or `e`/`E`
2. If found: extract substring, call libc `strtod()` (declare as extern in json.sans) to parse as double. Bitcast double to i64 for storage using a union-style trick: `store_double(buf, val)` then `load64(buf)` where buf is a temporary 8-byte alloc. Add helper functions `sans_double_to_bits(d:F) I` and `sans_bits_to_double(i:I) F` in a new `runtime/float_util.sans` (or inline in json.sans). Store as node type 6 (float)
3. If not found: keep existing atoi path (type 2 = integer)
4. Add `sans_json_get_float(node)` accessor that reads the i64 and bitcasts back to double
5. Update JSON serialization (`sans_json_stringify`) to handle type 6 — emit with decimal notation

**New JSON node type:** Tag 6 = float (add to existing: 0=null, 1=bool, 2=int, 3=string, 4=array, 5=object)

**Test:** Parse `{"pi": 3.14, "e": 2.718}`, verify float values are accessible and round-trip through stringify.

### 3c. Closure Capture Limit

**File:** `compiler/codegen.sans`, lines 930-1111
**Problem:** `emit_closure_aware_call1/2/3` hard-code branches for 0, 1, or 2 captures. 3+ captures silently miscompile.

**Fix — switch-based dispatch:**
Generate a switch on `ncaps` with cases 0 through 8. Each case loads the appropriate number of captures from the closure object and builds the call with `ncaps + explicit_args` parameters.

```
; pseudocode for emit_closure_aware_call1(fp, arg):
switch i64 ncaps, label %cap_err [
  i64 0, label %cap0
  i64 1, label %cap1
  i64 2, label %cap2
  ...
  i64 8, label %cap8
]
cap0: call fn_ptr(arg)
cap1: call fn_ptr(cap0, arg)
cap2: call fn_ptr(cap0, cap1, arg)
...
cap8: call fn_ptr(cap0..cap7, arg)
cap_err: call @sans_panic("closure: too many captures (max 8)")
```

Cap at 8 captures. Emit a clear compiler error if a closure captures more (add check in IR lowering, `ir.sans` ~line 3219).

**Test:** Closure capturing 3, 4, 5 variables passed to `map()` and `filter()`.

### 3d. Stack Size Portability

**File:** `compiler/main.sans`, line 457
**Problem:** `-Wl,-stack_size,0x10000000` is macOS Mach-O only. Linux gets 8MB default.

**Fix:** The link command in `main.sans` line 457 is currently a single hardcoded string with macOS-only flags (`-Wl,-stack_size`, `-Wl,-headerpad_max_install_names`, Homebrew rpaths via `brew --prefix`). There is no platform detection. The fix requires:

1. **Add platform detection:** Run `uname -s` via `sh()` at link time (or check a compile-time flag) to determine macOS vs Linux
2. **Branch the entire link command**, not just stack size:
   - macOS: keep current flags (`-Wl,-stack_size,0x10000000`, `-Wl,-headerpad_max_install_names`, Homebrew rpaths)
   - Linux: use `-Wl,-z,stacksize=268435456`, system pkg-config paths, no Homebrew rpaths
3. **Shared flags:** `-lcurl -lz -lssl -lcrypto` are common to both platforms

**Test:** Validated by Linux CI running the test suite (section 4). Add a recursive test fixture that exceeds 8MB stack to catch regressions.

### Files Changed
- `runtime/math.sans` — random fix
- `runtime/json.sans` — float parsing + new node type + serialization
- `compiler/codegen.sans` — closure capture dispatch expansion
- `compiler/ir.sans` — capture count validation
- `compiler/main.sans` — Linux stack size flag

## 4. Linux CI

### Design
Add `.github/workflows/test.yml` triggered on push to main and all PRs.

**Matrix:** `macos-15` (ARM64) + `ubuntu-latest` (x86_64)

**Steps per platform:**
1. Install LLVM 17 (`brew install llvm@17` on macOS, `apt install` on Ubuntu)
2. Download bootstrap compiler from latest release
3. Build compiler from source using bootstrap
4. Run `bash tests/run_tests.sh`

**Dependencies:** Needs the stack size fix (3d) to pass recursive tests on Linux. Needs runtime dependencies (libcurl, openssl, zlib) installed in the Ubuntu runner.

**Scope:** Build + test only. Release workflow unchanged.

### Files Changed
- `.github/workflows/test.yml` — new file
