# v0.8.x — Security Hardening & Adoption

**Theme:** Make Sans safe enough to run production services handling untrusted input, then ship adoption infrastructure.

**Strategy:** Security first (v0.8.0–v0.8.3), adoption second (v0.8.4–v0.8.5). Front-load crash-causing and memory-safety bugs, then build the platform that invites people to use Sans.

**Deferred to later milestones:**
- Dependency supply chain (lock files, `sans pkg audit`, integrity verification) — package manager is too basic; harden it when it matures
- Package manager input validation (`sans.json` schema validation, malicious git URL rejection) — deferred alongside supply chain work
- Path traversal prevention in file I/O APIs — deferred until file API surface stabilizes
- `sh()`/`system()` security documentation — deferred to v0.8.5 security guide
- Playground implementation approach (server-side vs WASM) — decided during v0.8.4 design phase

---

## v0.8.0 — Stop the Crashes

**Goal:** Eliminate every way Sans can silently crash or segfault today.

### Fuzz Testing Infrastructure
- Build a fuzzer harness that feeds random/malformed `.sans` files to the lexer and parser
- Compiler must return an error message for every malformed input — never panic or segfault
- Integrate into CI (GitHub Actions workflow)
- Success: 10M+ iterations with zero crashes

### SIGPIPE Handler
- Install `SIG_IGN` for SIGPIPE in HTTP server startup
- Currently a client disconnect kills the entire server process

### Panic Recovery Per Request
- `unwrap` (`!`) on error calls `exit(1)`, killing the whole server
- Add per-request error boundary in the worker loop using `setjmp`/`longjmp`: `setjmp` before handler dispatch, unwrap failures `longjmp` back instead of calling `exit(1)`
- Worker loop catches the jump, returns HTTP 500, keeps serving other connections
- Alternative if `setjmp`/`longjmp` proves problematic: redesign unwrap codegen to return an error value up the call stack instead of exiting

### Array/String Bounds Checking
- Array index operations emit a bounds check before the access
- String index/slice operations emit a bounds check
- Out-of-bounds produces a clear runtime error message instead of segfault

### Success Criteria
- Fuzz suite runs in CI, zero crashes after 10M iterations on lexer/parser
- SIGPIPE on client disconnect does not crash the server
- Single handler `unwrap` failure returns 500, server stays up
- `a[100]` on a 3-element array prints an error, not a segfault

## v0.8.1 — Memory Correctness

**Goal:** Fix known gaps in scope GC and audit memory safety at system boundaries.

### `scope_should_keep` Full Reference Walk
- Currently only preserves the direct return value and immediate array elements
- Fix to walk the full reference tree: map values, JSON children, nested containers, struct fields containing heap pointers
- All nested heap data reachable from a return value gets promoted to the caller scope

### Memory Audit
- Audit every extern function call boundary (libc, libcurl, libssl, zlib) for alloc/free balance
- Audit scope GC for use-after-free: returned closures referencing freed stack data, cross-thread sharing
- Audit HTTP server buffer handling (request parsing buffers, header storage, body accumulation) for overflows and off-by-one errors
- Document findings — fix what's fixable, document known limitations for the rest

### JSON Depth Limit
- Add configurable max recursion depth to the JSON parser (default 512)
- Return error instead of stack overflow on deeply nested input

### JSON Parse Errors as Result
- Parse failure currently returns `null`, indistinguishable from literal `null`
- Change `sans_json_parse` to return `Result<JsonValue>` instead
- **Breaking change:** Existing code using `json_parse()` will need to unwrap or handle the Result. Acceptable as a pre-1.0 break — no deprecation period needed, but document migration in release notes

### Success Criteria
- Returning a map/JSON/nested container from a function does not use-after-free
- Memory audit report committed to `docs/`
- `json_parse('{{{{{{...512 deep...')` returns an error, not a stack overflow
- `json_parse('invalid')` returns `Err`, not `null`

## v0.8.2 — Server Production-Ready

**Goal:** Make the HTTP server suitable for handling real traffic.

### Thread Pool with Bounded Workers
- Replace unbounded `spawn` per connection with a configurable worker pool (default 256)
- Reject connections beyond the limit with HTTP 503

### Request Timeouts
- Configurable read timeout (default 30s) and keep-alive timeout (default 60s)
- Currently threads pin forever on slow/stalled clients

### Multi-Read Request Assembly
- Single `srecv` call currently produces garbage on partial TCP reads
- Read until `\r\n\r\n` for headers, then read Content-Length bytes for body

### Graceful Shutdown
- On SIGTERM, stop accepting new connections
- Wait for in-flight workers to finish (with configurable timeout), then exit

### Input Validation
- Request size limits (default 1MB body)
- Header count limits (default 100 headers)
- URL length limits (default 8KB)
- Enforce the timeouts from above

### Success Criteria
- 257th concurrent connection gets 503 (with default pool size 256)
- Stalled client is disconnected after 30s
- Partial TCP read assembles correct full request
- `kill -TERM <pid>` drains in-flight requests then exits cleanly
- Oversized request gets 413, too many headers gets 431

## v0.8.3 — Data Correctness

**Goal:** Fix data handling bugs and extend fuzzing to the runtime layer.

### JSON Float Support
- `sans_json_p_number` currently truncates decimals/exponents to integers
- Fix to handle fractional and exponent parts, store as float when needed

### SipHash for Map and JSON
- Replace djb2 with SipHash (or similar keyed hash) in both `map.sans` and `json.sans`
- djb2 is trivially attackable for O(n^2) hash collision DoS
- Success: 10K crafted djb2-collision keys must complete in under 1s with the new hash (demonstrating O(n) behavior, not the O(n^2) that djb2 exhibits)

### Fuzz the Runtime
- Fuzz JSON parser with malformed JSON input
- Fuzz HTTP server with malformed requests (headers, body, WebSocket frames)
- Fuzz string operations with adversarial input (null bytes, unicode edge cases, extremely long strings)
- Add to the CI fuzzing workflow established in v0.8.0

### Success Criteria
- `json_parse('3.14')` returns a float, not 3
- `json_parse('1e10')` returns 10000000000, not 1
- 10K crafted djb2-collision keys complete in under 1s with the new hash (O(n), not O(n^2))
- Fuzz suite covers JSON parser, HTTP server, and string operations with zero crashes

## v0.8.4 — Adoption Platform

**Goal:** Ship the infrastructure that lets people try Sans without installing LLVM.

### Web Playground
- Sandboxed environment that compiles and runs Sans code, returns output
- Shareable links for code snippets (public by default — doubles as training data)
- "Try this example" buttons embedded in documentation pages
- Implementation approach (server-side vs WASM) decided during this release's design phase
- Track first-compile success rate: log whether AI-generated Sans compiles on first try, establish baseline comparison against Go/Python/TypeScript playgrounds

### Versioned ai-reference.md
- Add version number and changelog delta at the top of `ai-reference.md`
- CI tags a snapshot with each release

### llms.txt
- Publish `sans-lang.org/llms.txt` following the emerging convention
- Contents: current version, link to ai-reference.md, link to changelog, link to playground
- CI auto-updates on tag push

### Website Messaging Update
- New pitch: "The language AI gets right on the first try"
- Lead with: correctness, simplicity, batteries included, speed
- Drop: token count comparisons, cost savings claims
- Update `website/index.html` hero/tagline and `README.md` opening

### Success Criteria
- Playground live at a public URL, shareable links work
- `llms.txt` accessible at `sans-lang.org/llms.txt`
- `ai-reference.md` includes version number, auto-updated by CI
- Website hero text reflects correctness-first messaging
- First-compile success rate baseline measured and logged
