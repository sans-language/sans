# v0.8.4 — Adoption Platform Design

**Goal:** Ship the infrastructure that lets people try Sans without installing LLVM, and update messaging to lead with correctness.

**Decisions made during design:**
- Playground: Hybrid approach — static frontend on GitHub Pages, compilation API on DigitalOcean
- Sandboxing: Docker containers with resource limits
- Shareable links: Server-side SQLite storage with short IDs
- "Try this example" buttons: Website docs only (not markdown docs)
- Benchmarks page: Keep but reframe as "Comparisons"
- Homepage lead message: "The language AI gets right on the first try"
- Proof point: Mini-playground on homepage + "small language" explanation

---

## 1. Playground Server

### Stack
- **Language:** Go (single binary deploy)
- **Reverse proxy:** Nginx with TLS via Let's Encrypt at `api.sans.dev`
- **Storage:** SQLite (single file, zero config)
- **Sandboxing:** Docker (pre-built image with LLVM 17 + Sans compiler)
- **Host:** DigitalOcean droplet

### API Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `POST /api/run` | POST | Accept Sans code, compile + run in Docker, return stdout/stderr |
| `POST /api/share` | POST | Save code to SQLite, return short ID |
| `GET /api/snippet/:id` | GET | Retrieve saved code by ID |
| `GET /api/health` | GET | Health check |

### POST /api/run

**Request:**
```json
{"code": "p('hello')"}
```

**Response:**
```json
{"stdout": "hello\n", "stderr": "", "exit_code": 0, "compile_success": true}
```

**Error (compile failure):**
```json
{"stdout": "", "stderr": "error: unknown function 'foo' at line 1", "exit_code": 1, "compile_success": false}
```

The `compile_success` field enables tracking first-compile success rate.

### POST /api/share

**Request:**
```json
{"code": "p('hello')"}
```

**Response:**
```json
{"id": "aB3kQ7xm"}
```

### GET /api/snippet/:id

**Response:**
```json
{"id": "aB3kQ7xm", "code": "p('hello')"}
```

**404:**
```json
{"error": "snippet not found"}
```

### Docker Sandbox

- Pre-built image: Ubuntu base + LLVM 17 + latest Sans compiler binary
- Per-request execution:
  1. Write user code to a temp directory
  2. `docker run --rm --network=none --memory=256m --cpus=1 --pids-limit=64` with 10s timeout
  3. Inside container: `sans build /tmp/code.sans && /tmp/code`
  4. Capture stdout + stderr, return to client
  5. Clean up temp directory
- No network access inside container (prevents abuse, exfiltration)
- 10s hard timeout (kills container if exceeded)
- 256MB memory limit, 1 CPU, 64 process limit

### Rate Limiting

- Nginx-level rate limiting:
  - `/api/run`: 10 requests/minute per IP
  - `/api/share`: 10 requests/minute per IP
  - `/api/snippet`: 30 requests/minute per IP
- Returns HTTP 429 when exceeded

### Snippet Storage

- SQLite table schema:
  ```sql
  CREATE TABLE snippets (
      id TEXT PRIMARY KEY,
      code TEXT NOT NULL,
      created_at INTEGER NOT NULL
  );
  ```
- IDs: 8-character base62 (`[a-zA-Z0-9]`) — ~218 trillion combinations
- No expiration for alpha
- No size limit beyond the 1MB request body limit from v0.8.2

### First-Compile Success Rate Tracking

- Log each `/api/run` request: timestamp, code length, `compile_success` boolean
- SQLite table:
  ```sql
  CREATE TABLE compile_logs (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      code_length INTEGER NOT NULL,
      compile_success BOOLEAN NOT NULL,
      created_at INTEGER NOT NULL
  );
  ```
- Weekly aggregation query to compute success rate baseline
- No PII stored — code content is not logged, only length and outcome

## 2. Playground Frontend

### Location
- Full playground: `sans.dev/play` (new page in `website/play/index.html`)
- Homepage mini-playground: embedded in `website/index.html` hero section

### Components

**Code Editor:**
- CodeMirror 6 (lightweight, extensible, good mobile support)
- Sans syntax highlighting (basic — keywords, strings, numbers, comments)
- Dark/light theme matching the site's existing theme toggle

**Output Panel:**
- Below the editor
- Shows stdout (white/green), stderr (red), compilation errors (red)
- "Running..." spinner during execution

**Controls:**
- "Run" button (Ctrl+Enter shortcut) → POST `/api/run`
- "Share" button → POST `/api/share` → updates URL to `sans.dev/play/aB3kQ7xm`
- "Reset" button → clears editor to default example

**Shared Snippet Loading:**
- URL pattern: `sans.dev/play/:id`
- On page load, if `:id` present → GET `/api/snippet/:id` → populate editor
- 404 → show "Snippet not found" message, load default example

### Homepage Mini-Playground

- Compact version embedded in the hero section, below the tagline
- Pre-loaded with a simple example (e.g., HTTP server in 10 lines)
- Same Run button + output panel, smaller layout
- "Open in Playground" link to expand to full playground page

### "Try This Example" Buttons

- Added to `website/docs/index.html` only
- Small "Try it" button next to code examples in the documentation
- Clicking opens `sans.dev/play` with the example code pre-loaded (via POST to `/api/share`, then redirect)

## 6. Deployment & Infrastructure

### DigitalOcean Droplet Setup
- Ubuntu 22.04 LTS
- Docker installed
- Nginx with Let's Encrypt (certbot)
- Go binary deployed via scp or rsync
- SQLite database at `/var/lib/sans-playground/playground.db`
- Sans compiler Docker image pre-built and tagged

### DNS
- `api.sans.dev` → A record pointing to DO droplet IP
- `sans.dev` → continues pointing to GitHub Pages

### Monitoring
- Nginx access/error logs
- Go service logs to stdout (captured by systemd journal)
- Health check endpoint for uptime monitoring
