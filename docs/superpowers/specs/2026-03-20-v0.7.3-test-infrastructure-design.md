## 1. `sans test` Framework (Phase 1)

### Problem
Tests use a bash script that compiles each fixture, runs it, and checks exit codes. No assertions, no stdout validation, no test discovery — everything manually registered.

### Design

**Test file convention:**
- Files named `*_test.sans` are test files
- Functions named `test_*` are test functions
- Test functions take no arguments and return I (0 = pass, non-zero = fail)

**Example:**
```
import "assert"

test_addition() I {
  assert_eq(2 + 2, 4)
  assert(true)
  0
}

test_string_concat() I {
  s = "hello " + "world"
  assert_eq(s, "hello world")
  0
}
```

**Built-in assertions (compiler-provided builtins, not runtime):**

Assertions are implemented as **compiler builtins** (like `print`), not as generic runtime functions. This avoids the need for monomorphized generics in the runtime. The compiler knows the types of both arguments at typeck time and can emit the appropriate comparison and formatting code inline.

- `assert(cond:B)` — if false, print "assertion failed at file:line" and exit test
- `assert_eq(a, b)` — compiler checks types match, emits type-appropriate comparison. If a != b, print "expected {a}, got {b} at file:line". Supports I, S, B, F.
- `assert_ne(a, b)` — same as above but inverted
- `assert_err(r:R<T>)` — check result tag == 1
- `assert_ok(r:R<T>)` — check result tag == 0
- `assert_some(o:O<T>)` — check option tag == 1
- `assert_none(o:O<T>)` — check option tag == 0

Source location in assertion failures uses the source location propagation from v0.7.0 — the compiler embeds file:line as string constants at each assert call site.

**`sans test` command:**
1. Discover all `*_test.sans` files in current directory (recursive)
2. For each test file: compile it, discovering all `test_*` functions
3. Generate a test harness main() that calls each test function, collects results
4. Compile and run the harness
5. Print results:
   ```
   Running 12 tests...
   ✓ test_addition
   ✓ test_string_concat
   ✗ test_map_get: assertion failed at map_test.sans:15: expected 42, got 0

   11 passed, 1 failed
   ```
6. Exit 0 if all pass, exit 1 if any fail

**Implementation in compiler/main.sans:**
- Add `test` subcommand to CLI parser
- Test discovery: glob for `*_test.sans`, parse each, extract `test_*` function names
- Harness generation: emit a synthetic main() that calls each test function and tallies results
- Compile and run the harness as a single binary

### Files Changed
- `compiler/main.sans` — `sans test` subcommand, test discovery, harness generation
- `runtime/assert.sans` — new runtime module with assertion functions

### Migration
- Keep `tests/run_tests.sh` working alongside `sans test` during transition
- Gradually migrate fixtures to `*_test.sans` format
- Bash runner remains for tests that need exit-code validation or special compilation flags

## 3. Negative Test Infrastructure

### Problem
No way to test that invalid code correctly fails to compile with the expected error message.

### Design

**Convention:** Files named `*_neg.sans` or in a `tests/negative/` directory are negative tests. They should NOT compile successfully.

**Expected error annotation:**
```
// expect-error: type mismatch
x:I = "hello"
```

The `// expect-error: <substring>` comment annotates the **next non-comment line** — that is where the error is expected to occur. Multiple `// expect-error` annotations per file are supported (one per expected error), which aligns with the multi-error diagnostics from v0.7.0. The test runner compiles the file, expects it to fail, and checks that stderr output contains each expected substring.

**Test runner integration:**
- `sans test` handles negative tests: compile, expect non-zero exit, grep output for expected substring
- `run_tests.sh` can also support negative tests with a different registration function:
  ```bash
  run_negative_test "neg_type_mismatch" "tests/negative/type_mismatch.sans" "type mismatch"
  ```

**Initial negative tests (20+):**
- Type mismatches (assign string to int, wrong argument types)
- Unknown variable/function references
- Duplicate definitions
- Invalid syntax (missing braces, unclosed strings)
- Visibility violations (accessing private function from another module)
- Invalid pattern matching (wrong number of fields)
- Circular imports

### Files Changed
- `compiler/main.sans` — negative test support in `sans test`
- `tests/run_tests.sh` — negative test registration function
- `tests/negative/` — new directory with negative test fixtures

## 4. Compatibility Test Suite

### Problem
No mechanism to prevent breaking changes in future versions. Need a frozen set of programs that must compile and run identically across versions.

### Design

**Location:** `tests/compat/` — separate from regular fixtures

**Contents:** A curated set of Sans programs that exercise every stable language feature. These are frozen — once added, they must pass on all future versions.

**Categories:**
- `tests/compat/types.sans` — all basic types and operations
- `tests/compat/structs.sans` — struct creation, field access, methods
- `tests/compat/enums.sans` — enum variants, pattern matching
- `tests/compat/generics.sans` — generic structs, monomorphization
- `tests/compat/closures.sans` — lambda creation, captures, higher-order functions
- `tests/compat/result.sans` — Result creation, ?, !, combinators
- `tests/compat/option.sans` — Option creation, matching, chaining
- `tests/compat/channels.sans` — channel create, send, recv, select
- `tests/compat/modules.sans` — import, qualified access
- `tests/compat/collections.sans` — Array and Map operations

**Output verification:** Each compat test has a companion `.expected` file (e.g., `tests/compat/types.expected`) containing the expected exit code and stdout output. The runner compiles, executes, and compares both exit code and stdout against the expected file. Format:
```
exit: 0
stdout:
hello world
42
```

**Runner:** Compat tests are run by `tests/run_tests.sh` (not `sans test`), because they validate compilation + execution behavior, not assertions. They run as a separate section in the test runner. Any compat test failure blocks the release.

**Versioning:** Each compat test has a comment `// since: 0.7.3` indicating when it was frozen. New compat tests can be added in future versions but existing ones cannot be modified.

### Files Changed
- `tests/compat/` — new directory with curated test programs
- `tests/run_tests.sh` — compat test section

## 5. Self-Hosting Stability Investigation

### Problem
The compiler requires the v0.4.0 bootstrap binary because scope GC breaks self-compilation. This is documented but not well understood.

### Design

**Investigation tasks (not implementation):**
1. Build the compiler with scope GC enabled and catalog every failure
2. For each failure, create a minimal reproducing test case in `tests/fixtures/`
3. Classify each failure: fixable in 0.7.x scope GC hardening? Requires fundamental change?
4. Document findings in `docs/superpowers/specs/self-hosting-scope-gc-analysis.md`
5. If any failures were fixed by 0.7.2 scope GC hardening, attempt self-compilation with newer bootstrap

**Goal:** Reduce bootstrap dependency if possible. At minimum, understand exactly what breaks and why.

**Policy:** Tag a new bootstrap binary at each minor release (v0.7.0, v0.7.1, etc.) so the bootstrap chain stays current.

### Files Changed
- `docs/superpowers/specs/self-hosting-scope-gc-analysis.md` — new analysis document
- Possibly `compiler/main.sans` — updated bootstrap version reference
