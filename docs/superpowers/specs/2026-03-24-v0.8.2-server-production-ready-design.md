# v0.8.2 — Server Production-Ready

**Goal:** Make the Sans HTTP server suitable for handling real traffic by adding bounded concurrency, request assembly, timeouts, graceful shutdown, and input validation.

**Approach:** All configuration uses hardcoded defaults with optional global setter functions. The `serve(port handler)` and `serve_tls(port cert key handler)` APIs remain unchanged.

---

## Configuration Model

Global mutable state with setter builtins. Setters must be called before `serve()`/`serve_tls()`.

| Setting | Default | Setter | Error Response |
|---------|---------|--------|----------------|
| Max workers | 256 | `set_max_workers(n)` | 503 |
| Read timeout | 30s | `set_read_timeout(s)` | close |
| Keep-alive timeout | 60s | `set_keepalive_timeout(s)` | close |
| Drain timeout | 5s | `set_drain_timeout(s)` | — |
| Max body | 1048576 (1MB) | `set_max_body(n)` | 413 |
| Max header size | 8192 (8KB) | `set_max_headers(n)` | 431 |
| Max header count | 100 | `set_max_header_count(n)` | 431 |
| Max URL length | 8192 (8KB) | `set_max_url(n)` | 414 |

### Runtime Globals (server.sans)

```
g sans_max_workers = 256
g sans_active_workers = 0
g sans_read_timeout_ms = 30000
g sans_keepalive_timeout_ms = 60000
g sans_drain_timeout_ms = 5000
g sans_max_body_size = 1048576
g sans_max_header_size = 8192
g sans_max_header_count = 100
g sans_max_url_len = 8192
```

A pthread mutex protects `sans_active_workers`:
```
g sans_worker_mutex = 0   // initialized once in serve()/serve_tls()
```

### Setter Functions

Each setter is a new builtin in the compiler pipeline (typeck -> IR -> codegen) that writes to the corresponding global. Example:

```
set_max_workers(n:I) I { sans_max_workers = n; 0 }
```

These are implemented as runtime functions in `server.sans`, exposed as builtins via `typeck.sans` dispatch.

## 1. Thread Pool with Bounded Workers

### Changes to Accept Loop (sans_serve_accept_loop / sans_serve_tls_loop)

Before spawning a worker:
1. `pthread_mutex_lock(sans_worker_mutex)`
2. Check `sans_active_workers >= sans_max_workers`
3. If at limit: `pthread_mutex_unlock`, send 503 response on the accepted fd, close fd
4. If under limit: `sans_active_workers += 1`, `pthread_mutex_unlock`, spawn worker

### Worker Exit

At the end of `sans_serve_worker` / `sans_ssl_serve_worker`:
1. `pthread_mutex_lock(sans_worker_mutex)`
2. `sans_active_workers -= 1`
3. `pthread_mutex_unlock(sans_worker_mutex)`

### 503 Response

A minimal helper `sans_reject_503(fd)` that writes a hardcoded `HTTP/1.1 503 Service Unavailable\r\nContent-Length: 0\r\nConnection: close\r\n\r\n` to the fd and closes it. Same for SSL variant using `ssl_write`.

### Mutex Initialization

`sans_serve` and `sans_serve_tls` each call a one-time init:
```
sans_init_worker_mutex() I {
  if sans_worker_mutex == 0 {
    sans_worker_mutex = alloc(64)  // pthread_mutex_t size
    pthread_mutex_init(sans_worker_mutex, 0)
  }
  0
}
```

The runtime already has `pthread_mutex_init/lock/unlock` declared as LLVM externs.

## 2. Multi-Read Request Assembly

Replace single-read pattern with two-phase assembly.

### Phase 1: Header Assembly

```
sans_read_headers(fd) -> (buf, header_end, total_bytes) or error
```

1. Allocate 8KB buffer (sans_max_header_size)
2. Loop: `spoll(fd, sans_read_timeout_ms)` then `srecv(fd, buf + filled, remaining)`
3. After each read, scan for `\r\n\r\n` in the buffer
4. If found: header_end = position of `\r\n\r\n`, break
5. If buffer full without finding `\r\n\r\n`: send 431, close
6. If `spoll` returns 0 (timeout): close
7. If `srecv` returns <= 0: close

SSL variant uses `ssl_read` instead of `srecv`.

### Phase 2: Body Assembly

1. Parse `Content-Length` from headers (scan for `content-length:` in the header bytes)
2. If no Content-Length: body = whatever bytes are already in the buffer after `\r\n\r\n`
3. If Content-Length > sans_max_body_size: send 413, close
4. Calculate bytes_already_read = total_bytes - (header_end + 4)
5. If bytes_already_read >= Content-Length: done, slice the body
6. Otherwise: allocate body buffer, copy what we have, loop reading remainder with spoll+srecv until complete or timeout

### Affected Functions

- `sans_http_accept_read(cfd)` — plain HTTP initial read
- `sans_https_read(ssl, cfd)` — HTTPS initial read
- `sans_read_next_request_plain(fd)` — keep-alive next request (plain)
- `sans_ssl_read_next_plain(fd)` — keep-alive next request (plain, ssl module)
- `sans_ssl_read_next_ssl(fd, ssl)` — keep-alive next request (SSL)

All converge on the same two-phase pattern. Factor into shared helpers where possible. The SSL module duplicates much of server.sans — apply the same pattern to both.

### URL Length Validation

After parsing the request line (method + path), check `plen > sans_max_url_len`. If exceeded, send 414, close.

### Header Count Validation

In `sans_hdrs_add` / `sans_ssl_hdrs_add`, when `count >= sans_max_header_count`, send 431 instead of silently dropping. Currently the header array has a fixed capacity of 64 — change to allocate based on `sans_max_header_count`.

## 3. Request Timeouts

### Read Timeout

Every `srecv`/`ssl_read` call in the multi-read loop is preceded by `spoll(fd, sans_read_timeout_ms)`. Already handled in section 2's assembly loop.

### Keep-Alive Timeout

In `sans_serve_worker_loop` / `sans_ssl_serve_worker_loop`, when waiting for the next request on a keep-alive connection, use `spoll(fd, sans_keepalive_timeout_ms)` instead of blocking on `srecv`. If timeout, close gracefully.

### SSL Note

For SSL connections, `spoll` checks the underlying fd. If SSL has buffered data internally, `spoll` may report "not ready" even though `ssl_read` would succeed. This is acceptable: the keep-alive timeout is long enough (60s) that this edge case is benign. The read timeout (30s) similarly provides adequate protection for the common case of stalled clients.

## 4. Graceful Shutdown

### Current Behavior

```
while signal_check() == 0 { ... accept loop ... }
sclose(sfd)
```

### New Behavior

```
while signal_check() == 0 { ... accept loop ... }
// Stop accepting — close server fd immediately
sclose(sfd)
// Drain: wait for in-flight workers
elapsed := 0
while sans_active_workers > 0 && elapsed < sans_drain_timeout_ms {
  usleep(100000)  // 100ms
  elapsed += 100
}
// Log if workers remain
if sans_active_workers > 0 {
  // print warning
}
```

Workers still running after drain timeout are abandoned — the process exits and the OS cleans up their connections. This is standard behavior for HTTP servers.

Applies to both `sans_serve_accept_loop` and `sans_serve_tls_loop`.

## 5. New Status Codes

Add to `sans_status_text`:
- 413 — Payload Too Large
- 414 — URI Too Long
- 431 — Request Header Fields Too Large

503 (Service Unavailable) already exists.

## Compiler Changes

### typeck.sans

Add type checking for 8 new setter builtins:
- `set_max_workers(I) -> I`
- `set_read_timeout(I) -> I`
- `set_keepalive_timeout(I) -> I`
- `set_drain_timeout(I) -> I`
- `set_max_body(I) -> I`
- `set_max_headers(I) -> I`
- `set_max_header_count(I) -> I`
- `set_max_url(I) -> I`

All take one Int, return Int (0).

### constants.sans / ir.sans / codegen.sans

Each setter needs an IR instruction constant, IR lowering (lower_call_1), and codegen that emits a call to the corresponding `sans_set_*` runtime function.

## Success Criteria

- 257th concurrent connection gets 503 (with default pool size 256)
- Stalled client is disconnected after 30s
- Partial TCP read assembles correct full request
- `kill -TERM <pid>` drains in-flight requests then exits cleanly
- Oversized request body gets 413
- Oversized URL gets 414
- Too many headers gets 431
