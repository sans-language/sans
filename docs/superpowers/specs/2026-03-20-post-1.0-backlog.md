# Sans Post-1.0 Backlog

**Date:** 2026-03-20
**Status:** Backlog — not planned for 1.0.0
**Context:** Items identified during 1.0.0 roadmap design and GoF architecture review that are too invasive, too complex, or not critical enough for 1.0.

---

## Language Features

### Async/Await and Event Loops
**What:** Async functions, `await` keyword, event loop runtime for non-blocking I/O.
**Why deferred:** Concurrency via spawn/channels covers 1.0 use cases. Async requires new IR instructions, type syntax (`async fn`, `Future<T>`), and deep scope GC integration. Would require rethinking the runtime threading model.
**Prerequisite:** Stable scope GC (v0.7), thread-local storage (v0.6.1 done), builtin registry (v0.11).

### Operator Overloading
**What:** User-defined `+`, `-`, `*`, `==`, `<`, `[]` etc. on custom types.
**Why deferred:** Not needed for core use cases. Requires the builtin registry (v0.11) to decouple operator dispatch from hardcoded compiler logic.
**Prerequisite:** Builtin registry (v0.11), trait objects (v0.7).

### Macros / Metaprogramming
**What:** Compile-time code generation, AST manipulation, custom derive.
**Why deferred:** Too complex to stabilize by 1.0. Current pipeline (lexer → parser → typeck → IR) has no macro expansion phase. Would require a new pipeline stage between parsing and typeck.
**Prerequisite:** Stable AST representation, source location propagation (v0.7).

### Generics Constraints / Where Clauses
**What:** `fn sort<T: Ord>(arr: [T])` — bounding generic type parameters by traits.
**Why deferred:** Monomorphization works without constraints. Adding constraints requires changes to the type representation structure (currently a 32-byte struct with no room for bounds).
**Prerequisite:** Trait objects (v0.7), stable type representation.

### Implicit Type Coercion
**What:** Automatic int-to-float promotion, string conversion, etc.
**Why deferred:** Explicit is safer. Coercion rules are notoriously hard to get right and create surprising behavior.

### Union Types
**What:** `type StringOrInt = S | I` — ad-hoc sum types without enum wrapper.
**Why deferred:** Enums cover the primary use cases. Union types require tagged representation and exhaustiveness checking in match.

### Default Trait Implementations
**What:** `trait Display { fn show(self) S { "unknown" } }` — methods with default bodies.
**Why deferred:** Traits work as-is for 1.0. Default impls require method resolution order (MRO) logic and interaction with trait objects.
**Note:** v0.7 trait design must not block this — leave room in the vtable layout.

### Higher-Kinded Types
**What:** Abstract over type constructors: `trait Functor<F> where F: Type → Type`.
**Why deferred:** Advanced type theory, not needed for 1.0. Would require a fundamentally different type representation.

### Type Variance Annotations
**What:** Covariance/contravariance for generic type parameters.
**Why deferred:** Monomorphization avoids variance issues entirely. Only matters if we add subtyping or trait object coercion rules.

## Compiler Architecture (GoF-Driven)

### IR Representation Overhaul
**What:** Replace integer-based IR opcodes (262 hardcoded cases dispatched via `if op == IR_CONST`) with an object-based instruction model where each IR instruction type has its own `emit()`, `optimize()`, and `validate()` methods.
**Why deferred:** Touching all 262 opcode dispatch sites across ir.sans and codegen.sans is too invasive for the 1.0 timeline. Current model works, just doesn't compose.
**GoF pattern:** Command pattern — IR instructions as objects with behavior.
**Impact if done:** Enables optimization pass composition, multiple codegen backends (WASM, GPU), IR validation, and instruction metadata.
**Estimated effort:** 4-6 weeks. High risk of regression.

### Codegen Decomposition
**What:** Split the 4,220 LOC `codegen.sans` into separate phases:
1. Register allocation
2. SSA form construction
3. LLVM IR text emission
4. Scope GC instrumentation
5. Struct field layout calculation

Currently all interleaved in a single mega-function.
**Why deferred:** Functionally correct as-is. Decomposition is a refactoring effort that doesn't add user-facing features.
**GoF pattern:** Single Responsibility — each phase should be its own module.
**Impact if done:** Enables swapping register allocators, adding new GC strategies, and targeting non-LLVM backends.
**Estimated effort:** 3-4 weeks.

### Compiler Pass/Plugin System
**What:** Register arbitrary transformation passes between pipeline stages. Currently the pipeline is hardcoded: `lexer → parser → typeck → IR → codegen`.
**Why deferred:** No user demand yet. Useful for optimization passes, custom linting rules, or code instrumentation, but the hardcoded pipeline works for 1.0.
**GoF pattern:** Chain of Responsibility — passes as composable pipeline stages.
**Prerequisite:** IR overhaul (above), stable AST representation.

### Object Pool for Incremental Compilation
**What:** Recycle memory allocations for unchanged modules instead of fresh allocations per compilation. Currently every compilation allocates new maps, arrays, and strings from scratch.
**Why deferred:** Incremental compilation (v0.11) is the prerequisite. Object pooling is an optimization on top of incremental compilation, not a standalone feature.
**GoF pattern:** Object Pool — reuse expensive objects.
**Prerequisite:** Incremental compilation (v0.11), module-level IR caching.

### IR Instruction Metadata
**What:** Attach metadata to IR instructions — source location, optimization hints, type annotations, validation invariants. Currently IR instructions are raw memory blobs at fixed byte offsets (`load64(inst)`, `load64(inst + 8)`).
**Why deferred:** Source location propagation (v0.7) gives us the most important metadata. Full metadata requires the IR representation overhaul (above).
**GoF pattern:** Decorator — enrich instructions with metadata without changing their core behavior.
**Note:** v0.7's source location work is a stepping stone toward this.

## Runtime Architecture

### Opaque Type Encapsulation
**What:** Currently all opaque types (JsonValue, HttpResponse, Result, Map) expose their memory layout. User code could theoretically `load64(result + 8)` to extract values directly. Should enforce access only through public methods.
**Why deferred:** No user has exploited this yet, and the API surface (methods) is well-defined. True encapsulation would require codegen changes to prevent direct field access on opaque types.
**GoF pattern:** Information Hiding.

### Multiple Codegen Backends
**What:** Target WASM, GPU compute, or custom backends in addition to LLVM.
**Why deferred:** LLVM covers all 1.0 platform targets. Additional backends are a 2.0+ concern.
**Prerequisite:** IR overhaul, codegen decomposition.

---

## Tooling

### Integrated Debugger
**What:** Step-through debugging in IDE (breakpoints, variable inspection, call stack).
**Why deferred:** DWARF debug info (v0.9) enables lldb/gdb usage. A custom debugger UI is a nice-to-have, not a 1.0 requirement.
**Prerequisite:** DWARF debug info (v0.9).

### `sans fix` Migration Tool
**What:** Automatic code migration between versions (like `go fix`). Rewrites deprecated syntax or API changes.
**Why deferred:** Not needed until breaking changes accumulate between major versions.

---

## Priority Order for Post-1.0

If picking from this backlog for 1.1/2.0, suggested order:

1. **Default trait implementations** — low risk, high user value
2. **Generics constraints** — enables safer generic code
3. **Builtin registry extensibility → operator overloading** — natural extension of v0.11 work
4. **IR overhaul + codegen decomposition** — unlocks everything else
5. **Async/await** — biggest user-facing feature, needs IR overhaul first
6. **Macros** — highest complexity, defer as long as possible
