# Sans 1.0.0 Roadmap

**Date:** 2026-03-20
**Status:** Approved
**Goal:** Ship Sans 1.0.0 as the best AI-generated general-purpose compiled language
**Timeline:** Ship when ready — milestones are sequenced by dependency, not calendar dates
**Compatibility model:** Semver-strict — 1.0 is stable, 2.0 can break things

---

## Design Principles: What Makes Sans Best for AI Generation

Every feature, syntax choice, and design decision in Sans must serve these principles. They apply across all application types — backend APIs, CLI tools, data pipelines, compilers, anything.

### 1. Correctness by Default
AI generates bugs. The language must catch them before runtime.
- Strong static type system catches type errors at compile time
- `Option<T>` eliminates null/sentinel value bugs — no more "return 0 for not-found"
- `Result<T>` with `?` forces error handling — can't accidentally ignore failures
- Bounds checking on arrays/strings — no silent memory corruption
- Immutable by default (`=`) — AI must explicitly opt into mutation (`:=`)

*LLMs hallucinate. Every bug the compiler catches is one the human doesn't have to find in review. The more the type system rejects, the more reliable AI-generated code is.*

### 2. One Obvious Way
AI generates inconsistent code when there are multiple ways to do the same thing.
- One string type, one integer type, one map type
- One error handling pattern (`Result<T>` + `?`)
- One concurrency model (spawn + channels + select)
- `sans fmt` enforces the one true style — AI output is always canonical
- Short form is canonical (`I`, `S`, `p`, `fr`). Long forms exist for discoverability but `sans fmt` normalizes to short form.

*LLMs generate more correct code when there's one idiomatic path. One canonical form per concept = more predictable, more consistent output across generations.*

### 3. Small Surface Area, Deep Composability
Fewer concepts that compose well beat many special-case features.
- Small set of types (I, F, S, B, Array, Map, Struct, Enum, Result, Option) that compose via generics
- Traits for shared behavior — one dispatch mechanism, not per-type special cases
- Closures/lambdas as first-class — pass behavior as data
- Fewer keywords to learn = fewer for AI to misuse

*LLMs perform better on languages with smaller grammars. Every additional keyword is another hallucination opportunity. Composable primitives let AI build complex behavior from simple, well-understood parts.*

### 4. Fast Feedback Loop
AI-generated code needs rapid validation. Compile fast, fail fast, fix fast.
- Fast compilation (target: self-compile in <5 seconds)
- Error messages with file:line:column, source context, and suggestions
- Multiple errors reported per compilation (don't stop at first)
- `sans test` runs in seconds, not minutes
- LSP gives instant feedback in-editor

*In an AI coding loop (generate → compile → fix → repeat), every second of compile time is wasted. Every unclear error message is a wasted retry. The faster and clearer the feedback, the fewer tokens burned on retries.*

### 5. Safe by Default, Unsafe by Choice
Production code generated by AI must not have security vulnerabilities by default.
- Memory safety via scope GC — no manual malloc/free for common patterns
- No implicit shell execution — `sh()` is explicit and documented as dangerous
- Bounds checking everywhere
- `defer` for guaranteed resource cleanup
- Thread safety via channels, not shared mutable state
- HTTP server with sane defaults (timeouts, size limits, thread pools)

*AI cannot reason about security. It will generate `sh(user_input)` if you let it. The language must make the safe path the easy path and the dangerous path explicit.*

### 6. Batteries Included
AI should not need to import external packages for common patterns.
- HTTP server/client built in
- JSON parsing/generation built in
- File I/O, crypto, regex, date/time, path manipulation in stdlib
- Test framework built in (`sans test`)
- Package manager built in (`sans pkg`)

*External dependencies are where AI generation breaks down — hallucinated package names, wrong API versions, packages that don't exist. Every built-in capability is one less hallucination opportunity.*

### 7. Dense, Scannable Output
Short form syntax is a feature, not a compromise. Code is read more than it's written, and density aids comprehension at scale.
- Short type names (`I`, `S`, `B`, `F`, `R<T>`) — more logic per line, less visual noise
- Expression bodies (`f(x:I) I = x*2`) — function fits on one line
- No commas in arrays (`[1 2 3]`) — fewer tokens, cleaner visual
- `sans fmt` normalizes everything — AI output is always consistent
- Density means more context fits in editor viewport and in LLM context windows

*Short form optimizes the full pipeline: AI generates fewer tokens (cheaper, faster), code reviews see more logic per screen (faster review), and LLM context windows fit more code (better reasoning about large codebases). The one-time learning cost amortizes across every generation and every review.*

## Current State (v0.6.1)

Sans is a self-hosted compiled language with:

- **Compiler:** ~15,700 LOC, fully self-hosted (lexer → parser → typeck → IR → codegen → LLVM)
- **Runtime:** 17 modules (~6,600 LOC) — arena, arrays, curl, functional, fs, http, json, log, map, math, process, rc, result, server, sock, ssl, string_ext
- **Language features:** Structs, enums, generics with monomorphization, traits, closures, pattern matching, Result with `?`/`!`, concurrency (spawn, channels, mutex)
- **HTTP server:** Auto-threading, keep-alive, gzip, WebSocket, CORS, streaming, static files, graceful shutdown
- **Package manager:** `sans pkg` with init, add, install, remove, list, update (git-based deps)
- **Tests:** 134 E2E fixtures, 107 registered tests
- **Editor:** VSCode extension with syntax highlighting and hover docs
- **CI/CD:** Automated releases for macOS ARM64 and Linux x86_64
- **Platforms:** macOS (ARM64), Linux (x86_64)

## Explicitly Deferred to Post-1.0

These features are **not** planned for 1.0.0 to prevent scope creep. See `docs/superpowers/specs/2026-03-20-post-1.0-backlog.md` for the full backlog with rationale.

**Language features:**
- **Async/await and event loops** — Concurrency via spawn/channels is sufficient for 1.0. Async I/O is a 2.0 consideration.
- **Operator overloading** — Not needed for core use cases.
- **Macros / metaprogramming** — Too complex to stabilize by 1.0.
- **Generics constraints / where clauses** — Monomorphization works without them for now.
- **Implicit type coercion** — Explicit is safer; revisit in 2.0.
- **Union types** — Enums cover the primary use cases.
- **Default trait implementations** — Traits work as-is for 1.0.
- **Higher-kinded types** — Not needed for 1.0 use cases; requires type system redesign.
- **Type variance annotations** — Monomorphization avoids the issue for now.

**Compiler architecture (GoF-driven):**
- **IR representation overhaul** — Replace integer-based opcode dispatch (262 cases) with object-based IR instructions with `emit()` behavior. Enables optimization pass composition and multiple backends. Too invasive for 1.0.
- **Codegen decomposition** — Split 4,220 LOC codegen into separate phases (register allocation, SSA form, LLVM text emission, scope GC instrumentation). Currently monolithic.
- **Compiler pass/plugin system** — Register arbitrary transformation passes between IR and codegen. Currently hardcoded pipeline.
- **Object pool for incremental compilation** — Recycle allocations for unchanged modules instead of fresh allocations per compilation. Needed for large codebase scaling.
- **IR instruction metadata** — Add source location, optimization hints, and validation invariants to IR instructions. Currently raw memory blobs at fixed offsets.

## Cross-Cutting Policies

### Bootstrap Chain Preservation
Every minor release **must** maintain a working bootstrap binary. Before any change that affects the compiler's own compilation, verify the bootstrap chain still works. CI must always be able to build the compiler from the most recent release binary.

### Backward Compatibility Testing
Starting in v0.7, maintain a **compatibility test suite** — a set of programs that must compile and run identically across all versions from that point forward. Any breaking change must be flagged and justified.

### Linux CI From Day One
The existing test suite must run on Linux in CI starting in v0.7, not v0.12. Platform-specific bugs caught early are 10x cheaper to fix.

### Memory Model: Scope GC is the 1.0 Model
Scope GC is the committed memory management model for 1.0. Known limitations (deeply nested structures in compiler self-compilation, performance overhead in hot loops) will be addressed through targeted fixes, not a replacement. ARC or other models are deferred to 2.0 if needed. The v0.6.1 thread-local fix (f80ca28) makes scope GC viable for concurrent workloads.

## v0.7 — Stability & Correctness

**Theme:** Make what exists rock-solid. Fix every compiler edge case, improve error messages, and dramatically expand test coverage.

### Source Location Propagation (GoF: enables Observer pattern for diagnostics)
Currently source location (file, line, column) is lost after parsing. Thread it through the entire pipeline:
- Lexer: tokens carry file:line:column
- Parser: AST nodes carry source spans
- Typeck: errors reference source spans, not just message strings
- IR: instructions carry source location for debug info and error reporting
- Codegen: source locations emitted as LLVM debug metadata (prep for v0.9 DWARF)
This is foundational — LSP, linting, warnings, debugger, and error messages all depend on it.

### Structured Compiler Diagnostics (GoF: Observer pattern)
Replace the current `print(msg); exit(1)` error model with structured diagnostics:
- Diagnostic type with severity (error/warning/info), file:line:column, message, and optional suggestion
- Collect multiple errors before exiting (don't fail on first error)
- Show source context in error output (the offending line with a caret pointing to the error)
- Add warnings: unused variables, unreachable code, shadowed names
- Diagnostic sink abstraction — compiler, LSP, and linter all consume diagnostics through the same interface

### Language Gaps for 1.0
- **`Map<K, V>` generalization** — Currently maps only support string keys and integer values. Generalize to arbitrary key/value types. This is a breaking change and must land before the compatibility test suite freezes behavior.
- **Resource cleanup mechanism** — Add `defer` statement for cleanup of file handles, sockets, database connections. Scope GC handles memory; `defer` handles everything else. Design: `defer expr` executes `expr` when the enclosing function returns, in LIFO order (like Go's defer).
- **Trait objects / dynamic dispatch** — Enable polymorphic collections and the Strategy pattern. Without `dyn Trait` (or equivalent), you can't have a list of heterogeneous items implementing the same trait, which is table-stakes for a general-purpose language competing with Go's interfaces. Design: trait-typed variables hold a fat pointer (data ptr + vtable ptr), method calls dispatch through the vtable. Ensure the 1.0 trait design leaves room for 2.0 extensions (default implementations, trait inheritance, associated types) without breaking changes.
- **`Option<T>` type** — Sans currently uses sentinel values (0 for not-found, 0 for missing map keys). This is the billion-dollar mistake. Add `Option<T>` with `some(v)`, `none`, and integrate with `?` operator. `Map.get()` should return `Option<V>`, `Array.find()` should return `Option<T>`. Breaking change — must land before compatibility freeze.
- **Channel `select` statement** — Cannot multiplex channels without `select`. Cannot implement timeouts, cancellation, or fan-in. Non-negotiable for real concurrent programs. Design: `select { ch1.recv() => ..., ch2.recv() => ..., timeout(ms) => ... }`.
- **Result combinators** — Add `result.map(fn)`, `result.and_then(fn)`, `result.map_err(fn)` for error handling without unwrapping into variables at every step.
- **Module system improvements** — Move forward from v0.10 since the LSP (v0.9) needs to understand the module system:
  - Namespaced imports (`import "http" as h` or selective imports)
  - Visibility modifiers — public vs private functions/structs within a module
  - Re-exports for building clean public APIs in packages

### Language Edge Cases & Bug Fixes
- Audit and fix generic monomorphization edge cases (deeply nested generics, recursive generic types)
- Fix pattern matching gaps — add struct/tuple destructuring in match arms
- Fix any known issues with enum variant data extraction in complex expressions
- Harden import system — circular import detection, better error messages for missing modules

### Scope GC Hardening (Detailed)
This is the most critical correctness work in the release. Known issues to address:
- **Closures capturing mutable state:** When a closure captures a variable that is freed by scope GC in the enclosing function, the closure holds a dangling pointer. Identify all patterns where this occurs and fix (promote captured allocations to closure scope).
- **Nested loops with early returns:** `break`/`continue`/`return` inside nested loops may skip scope cleanup. Audit all control flow exits and ensure scope_exit runs.
- **Complex container nesting:** Arrays of maps of arrays, structs containing arrays of Results — verify that type-aware destructors walk the full tree.
- **Cross-function promotion edge cases:** When a function returns a value that contains nested heap allocations, all nested allocations must be promoted to the caller's scope. Test with 3+ levels of nesting.
- **Loop allocation leak:** Allocations inside `while true` loops accumulate until function return. For server handlers that loop, this means unbounded memory growth. Add per-iteration scope cleanup or loop-scoped arenas.
- **Self-compilation patterns:** Document the specific patterns in the compiler's own code that break under scope GC. Create minimal reproducing test cases. Fix what's fixable; document what's deferred.

### Runtime Deduplication (GoF: DRY)
`runtime/map.sans` and `runtime/json.sans` contain duplicated hash table implementations (same hashing algorithm, same linear probing logic). Extract a shared hash table module that both consume. Bug fixes currently need to land in two places.

### Dead Code Elimination (Moved from v0.11)
- Don't compile unused functions or imports — reduces binary size and improves compile times
- Constant folding — evaluate constant expressions at compile time
- These are correctness/hygiene features, not optimizations. They belong here.

### Test Coverage Expansion
- **Target: 250+ test fixtures** (from current 134)
- Add comprehensive tests for:
  - Package manager commands
  - File I/O operations
  - Process execution (`sh`, `system`)
  - HTTP server features (headers, cookies, query params, WebSocket)
  - Concurrency edge cases
  - Cross-module imports
  - Scope GC under stress
- Add negative tests — code that *should* fail to compile, with expected error messages
- Add a test runner mode that checks for expected compiler errors

### Basic Test Framework (`sans test` — Phase 1)
Moved forward from v0.10 because the test coverage expansion needs it:
- `sans test` command discovers and runs `*_test.sans` files
- Test functions prefixed with `test_` auto-discovered
- Basic assertions: `assert(cond)`, `assert_eq(a b)`
- Pass/fail counts, failure details with source location, CI-friendly exit codes
- Migrate existing bash test runner to use `sans test` internally

### Compiler Self-Hosting Stability
- Investigate whether newer compiler versions can self-compile without the v0.4.0 bootstrap
- Document or fix the specific scope GC patterns that break self-compilation
- Goal: reduce bootstrap dependency if possible
- **Policy:** Tag a new bootstrap binary at each minor release

### Success Criteria
- 250+ test fixtures passing
- Zero known compiler crashes from test suite and manual testing (fuzz verification deferred to v0.8)
- All error messages include file, line, and actionable description
- Negative test infrastructure in place
- `sans test` running and used for new test fixtures
- `Map<K, V>` works with arbitrary key/value types
- `defer` statement implemented
- `Option<T>` type with `some`/`none`, integrated with Map.get() and Array.find()
- Channel `select` with timeout support
- Result combinators (map, and_then, map_err)
- Namespaced imports and visibility modifiers working
- Trait objects / dynamic dispatch working (polymorphic collections, trait-typed variables)
- Dead code elimination functional
- Linux CI running full test suite
- Compatibility test suite established
- Source locations propagated through full pipeline (lexer → parser → typeck → IR)
- Compiler reports multiple errors before exiting
- Map/json hash table deduplicated into shared module

## v0.8 — Security Hardening

**Theme:** Make Sans safe enough to run production services handling untrusted input.

### Fuzz Testing Infrastructure
- Build a fuzzer for the lexer/parser — feed random/malformed `.sans` files, ensure the compiler never crashes (panics, segfaults, infinite loops)
- Fuzz the JSON parser with malformed JSON input
- Fuzz the HTTP server with malformed requests (headers, body, WebSocket frames)
- Fuzz string operations with adversarial input (null bytes, unicode edge cases, extremely long strings)

### HTTP Server Hardening (Critical — from production safety review)
These are blocking issues that make the server unsuitable for production:
- **Thread pool with bounded workers** — replace unbounded `spawn` per connection with a worker pool (configurable max threads, default 256). Reject connections beyond limit with 503.
- **SIGPIPE handler** — install `SIG_IGN` for SIGPIPE. Currently client disconnect crashes the server.
- **Request timeouts** — configurable read timeout (default 30s), keep-alive timeout (default 60s). Currently threads pin forever.
- **Multi-read request assembly** — single `srecv` call produces garbage on partial TCP reads. Read until `\r\n\r\n` for headers, then read Content-Length bytes for body.
- **Graceful shutdown drain** — on SIGTERM, stop accepting, wait for in-flight workers (with timeout), then exit.
- **Panic recovery per request** — `unwrap` (`!`) on error calls `exit(1)`, killing the whole server. Add per-request error boundary so one handler failure doesn't crash all connections. At minimum, catch unwrap failures in the worker loop and return 500.

### JSON Parser Hardening (Critical)
- **Float support** — JSON numbers with decimals/exponents are silently truncated to integers. Fix `sans_json_p_number` to handle fractional and exponent parts.
- **Depth limit** — recursive descent with no depth limit causes stack overflow on nested input. Add configurable max depth (default 512).
- **Parse error distinction** — parse failure returns `null`, indistinguishable from literal `null`. Return `Result<JsonValue>` instead.
- **Hash collision resistance** — replace djb2 with SipHash or similar keyed hash in both map.sans and json.sans. djb2 is trivially attackable for O(n^2) DoS.

### Memory Safety Audit
- Audit all system library call boundaries (libc, libcurl, libssl, zlib) — ensure every `alloc`/`free` pair is balanced across extern function calls
- Audit scope GC for use-after-free scenarios (returned closures referencing freed stack data, cross-thread sharing)
- Add bounds checking on array/string indexing with clear runtime error messages instead of segfaults
- Audit buffer handling in the HTTP server (request parsing, header limits, body size limits)
- **Audit scope_should_keep** — currently only preserves direct return value and immediate array elements. Map values, JSON children, nested containers, and struct fields containing heap pointers are NOT preserved. Fix to walk the full reference tree.

### Input Validation & Hardening
- HTTP server: request size limits, header count limits, URL length limits, timeout enforcement
- Package manager: validate `sans.json` schema, reject malicious git URLs, verify checksums
- File I/O: path traversal prevention in any path-accepting API
- Process execution: document security implications of `sh()` and `system()`, consider sandboxing options

### Dependency Supply Chain
- Package integrity verification (hash/checksum of downloaded packages)
- Lock file support (`sans.lock`) for reproducible builds
- Dependency audit command (`sans pkg audit`) — at minimum, list all transitive deps with their sources

### Security Documentation
- Security best practices guide for Sans applications
- Document the threat model for the package manager
- Responsible disclosure policy for the project itself

### Web Playground (Critical for adoption)
Highest-leverage single item for adoption. Eliminates the #1 onboarding barrier (LLVM 17 installation).
- Sandboxed backend that compiles and runs Sans code, returns output
- Shareable links for examples
- Embed in documentation pages ("try this example" buttons)
- Enables anyone to try Sans in 10 seconds from any platform, including Windows
- Implementation: either compile Sans to WASM for client-side execution, or run a sandboxed `sans build` + execute on the server

### Early Adopter Program
- Begin informal outreach — invite developers to try Sans and report issues
- Create a feedback channel (GitHub Discussions, Discord, or similar)
- Language design feedback at this stage is 10x more valuable than at RC stage
- Track pain points for prioritization in v0.9+

### Success Criteria
- Fuzz suite runs in CI with no crashes after 10M iterations
- All array/string indexing is bounds-checked
- HTTP server handles adversarial input without crashes (thread pool, timeouts, SIGPIPE, multi-read, graceful drain)
- JSON parser handles floats, deep nesting, and parse errors correctly
- Hash collision attack on map/JSON takes >1s for 10K keys (SipHash or equivalent)
- Single handler crash does not bring down the server
- `sans.lock` support implemented
- Security guide published

## v0.9 — Tooling (LSP, Formatter, Linter)

**Theme:** Build the developer experience that makes Sans feel like a modern, professional language.

### Language Server Protocol (LSP)

Build a Sans LSP server as a separate binary (`sans-lsp`), **written in Sans** (dogfooding — proves 1.0 readiness).

**Phase 1 (v0.9 — the 80% use case):**
- Inline diagnostics (compiler errors/warnings as you type)
- Hover information (type signatures, doc comments)
- Go-to-definition (functions, structs, enums, traits, imports)
- Signature help (function parameter hints)

**Phase 2 (v0.10 — alongside ecosystem work):**
- Find all references
- Autocomplete (keywords, in-scope variables, struct fields, methods, module exports)
- Rename symbol (local and cross-file)

**Architecture:** The LSP will need incremental/partial parsing — full recompilation on every keystroke won't scale. Build a lightweight analysis mode in the compiler that stops after typeck. This is essentially a second, partial compiler front-end — scope it as a major sub-project.

**Prerequisite:** Buffered I/O for JSON-RPC over stdio. If not already available, add buffered stdin/stdout reading to the runtime before starting LSP work.

**Fallback plan:** If writing the LSP in Sans hits blockers (scope GC instability for long-running processes, performance issues, missing language features), fall back to a minimal LSP in Go or TypeScript that shells out to `sans build --check` for diagnostics. Ship what works. Nobody cares what language the LSP is written in — they care that it works. Rewrite in Sans post-1.0 when the language is proven for this class of software.

### Formatter (`sans fmt`)
- Canonical code formatter — one true style, no configuration (like `gofmt`)
- Enforces AI-optimized syntax conventions (short type names, no commas in arrays, etc.)
- Idempotent — running it twice produces the same output
- Integrate into CI as a check

### DWARF Debug Info (GoF: enables Visitor pattern for runtime introspection)
Emit LLVM debug metadata using source locations propagated in v0.7:
- `!dbg` metadata on LLVM instructions mapping back to Sans source file:line
- Compile with `sans build --debug` to include full DWARF info
- Enables: stepping through Sans code in lldb/gdb, meaningful stack traces on crash, profiling with source attribution
- Prerequisite for any future debugger integration

### Linter (`sans lint`)
- Static analysis beyond what the compiler catches
- Rules: unused imports, unreachable code after return, empty catch blocks, shadowed variables in nested scopes, mutable variables never mutated
- Configurable severity (warn vs error)
- Shares analysis infrastructure with the LSP

### Multi-Editor Support
- **VSCode:** Update extension to use LSP (replace custom hover logic with LSP-powered features)
- **Neovim:** Plugin with LSP client config + syntax highlighting (tree-sitter or tmLanguage)
- **Emacs:** LSP client config + syntax highlighting
- **JetBrains:** Generic LSP plugin configuration + TextMate grammar import
- All editors get the same feature set via the shared LSP server

### Success Criteria
- LSP Phase 1: diagnostics, hover, go-to-definition, signature help working in all supported editors
- `sans fmt` formats all stdlib and test fixtures idempotently
- `sans lint` catches common issues with zero false positives on existing codebase
- VSCode, Neovim, Emacs, JetBrains all functional with LSP Phase 1
- `sans build --debug` produces binaries with DWARF info; lldb can show Sans source on crash

## v0.11 — Performance

**Theme:** Make Sans competitive with Go on benchmarks and ensure it stays fast as codebases grow.

### Compiler Performance
- Profile compilation of the compiler itself (the largest Sans codebase) — identify bottlenecks in lexer, parser, typeck, IR, codegen
- Incremental compilation — cache per-module compilation artifacts, only recompile changed modules and their dependents
- Parallel compilation — compile independent modules concurrently
- Target: self-compilation under 5 seconds (benchmark and track)

### Runtime Performance
- Optimize scope GC — reduce overhead of tracking/freeing allocations in hot loops
- String interning for small/repeated strings
- Array/map pre-allocation hints where sizes are known at compile time
- Optimize method dispatch — vtable or direct call when type is statically known
- Reduce unnecessary heap allocations (stack-allocate small structs, return value optimization)

### LLVM Optimization Passes
- Audit which optimization flags are passed to `llc`/`opt`
- Ensure `-O2` or equivalent is available as a build flag (`sans build --release`)
- Profile-guided optimization (PGO) support as a stretch goal

### Benchmark Suite
- Automated benchmark suite that runs on every release
- Categories: compute (fibonacci, sorting, matrix multiply), I/O (file read/write, HTTP throughput), concurrency (channel throughput, spawn overhead), string processing, JSON parse/stringify
- Compare against Go and Rust equivalents for the same benchmarks
- Publish results on the website (`sans-lang.org/benchmarks`)
- CI integration — fail the build if a benchmark regresses more than 10%

### Builtin Registry (GoF: Strategy/Factory pattern)
Extract 131 hardcoded `if name == "builtin"` chains in typeck.sans and corresponding IR/codegen dispatch into a lookup table:
- `Map<String, BuiltinDef>` where `BuiltinDef` = `{ param_types, return_type, ir_opcode, codegen_emitter }`
- Typeck, IR, and codegen all read from the same registry instead of maintaining parallel if-else chains
- Adding a new builtin becomes a single registry entry instead of 4-file surgery
- Prerequisite for any future extensibility (user-defined intrinsics, operator overloading in 2.0)
- Also improves compile time — table lookup vs. 131 string comparisons per call

## v0.12 — Platform Support (Windows, ARM Linux)

**Theme:** Sans builds and runs everywhere developers work.

### Windows Native Support
- Port the compiler to build on Windows — LLVM 17 via official installer or chocolatey
- Resolve platform-specific issues: path separators, line endings, temp directories, process spawning
- Runtime porting: replace POSIX-specific calls (socket, signal handling, file I/O) with cross-platform abstractions or conditional codegen
- HTTP server on Windows — Winsock instead of POSIX sockets
- TLS/SSL — link against Windows SChannel or ship with OpenSSL
- Package manager — ensure `git` availability on Windows or bundle `libgit2`
- Windows CI — GitHub Actions Windows runner, build + run full test suite
- Windows installer — `.msi` or standalone `.zip` with PATH setup instructions

### ARM Linux Support
- ARM64 (aarch64) Linux builds — for cloud servers (AWS Graviton, Oracle Ampere)
- CI builds for linux-aarch64
- Test on actual ARM hardware or QEMU in CI
- Cross-compilation support: build ARM binaries from x86 and vice versa

### Cross-Compilation
- `sans build --target=linux-x86_64`, `--target=linux-arm64`, `--target=windows-x86_64`, `--target=darwin-arm64`
- Leverage LLVM's built-in cross-compilation — Sans generates LLVM IR, so the backend supports this in theory. The work is in linking against the right system libraries for each target.

### Platform Abstraction Layer
- Abstract platform-specific runtime calls behind a common interface
- File paths, process spawning, networking, TLS, temp directories, environment variables
- Each platform implements the interface — avoids conditional sprawl in codegen

### Build Tags / Conditional Compilation
- Allow user code to encapsulate platform-specific behavior (e.g., `//sans:build windows` or `@platform("linux")`)
- Without this, library authors have no way to write cross-platform packages — the platform abstraction only exists at the runtime level
- Design should be simple (Go-style file suffix convention like `_windows.sans` or comment-based tags), not a preprocessor

### Package Manager Transport Abstraction
- Abstract package fetching behind a transport interface instead of shelling out to system `git`
- Embed git protocol support or use `libgit2` bindings so the package manager works without a system `git` installation
- Critical for Windows where `git` may not be on PATH

### Release Matrix
- GitHub releases ship binaries for: macOS ARM64, macOS x86_64, Linux x86_64, Linux ARM64, Windows x86_64
- Install script (`curl -fsSL https://sans-lang.org/install.sh | sh`) detects platform and downloads the right binary
- Windows: PowerShell install script equivalent

- Windows CI green

- Build tags / conditional compilation working
- Package manager fetches without requiring system `git`

## Skeptic Review Findings (2026-03-20)

Four parallel skeptic reviews were run: language design, execution risk, production safety, and developer adoption.

### Acknowledged Risks

**v0.7 scope is too large.** Independent reviewers estimate 6-9 months for one person, not 3. Trait objects alone could consume the budget. Mitigation: if v0.7 slips, consider splitting into v0.7a (stability fundamentals) and v0.7b (language features).

**The memory model has fundamental limits.** The compiler itself cannot use scope GC. Long-lived data structures, caches, and graph-structured data do not fit scope-based deallocation. This is accepted — scope GC targets request-handler-shaped workloads. Complex programs can use manual allocation. Document this honestly.

**LSP in Sans is high risk.** The LSP will be the first non-trivial, long-running, stateful application written in Sans. It will surface every remaining bug. Fallback plan added to v0.9.

### Strategic Questions (resolved)

1. **Positioning:** Reframed from "competing with Go/Rust" to "best AI-generated general-purpose language." This is a genuinely unclaimed niche. Sans is not narrowed to backends — it targets any application type (APIs, CLI tools, data pipelines, compilers) through AI generation. Full scope (Windows, stdlib, multi-editor, LSP) remains — a general-purpose language needs all of it.

2. **Syntax emphasis:** Short form is canonical. `sans fmt` normalizes to short form. Density optimizes the full AI pipeline: cheaper generation, more logic per screen in review, more code in LLM context windows. See Principle 7 (Dense, Scannable Output).

3. **Timeline:** Ship when ready. Milestones are sequenced by dependency, not calendar dates. Quality over speed — 1.0 means production-ready, not "on schedule."
