# v0.7.1 — Type System

**Date:** 2026-03-20
**Status:** Draft
**Goal:** Add Option<T>, generalize Map<K,V>, add distinct types for handle safety, Result combinators, and trait objects
**Depends on:** v0.7.0 (source locations + diagnostics for error reporting)

---

## Overview

v0.7.1 fills the type system gaps that block Sans from being a credible general-purpose language. The biggest wins are Option<T> (eliminates sentinel value bugs) and Map<K,V> generalization (currently string→int only). Trait objects enable polymorphic collections. Distinct types prevent handle confusion.

---

## 1. Option<T>

### Problem
Sans uses sentinel values: `Map.get()` returns 0 for missing keys, `Array.find()` returns 0 for not-found. This is the billion-dollar mistake — 0 is indistinguishable from a legitimate value.

### Design

**Type representation:**
- New type tag: `TY_OPTION` in constants.sans
- Layout: `make_option_type(inner:I)` — same pattern as Result<T>, Array<T>
- Short alias: `O<T>` for `Option<T>`

**Runtime representation (16 bytes):**
```
offset 0: tag (0 = none, 1 = some)
offset 8: value (if some)
```

**Runtime functions** (new `runtime/option.sans`):
- `sans_option_some(value:I) I` — alloc 16, set tag=1, store value
- `sans_option_none() I` — alloc 16, set tag=0
- `sans_option_is_some(opt:I) I` — load tag, return tag == 1
- `sans_option_is_none(opt:I) I` — load tag, return tag == 0
- `sans_option_unwrap(opt:I) I` — if none, panic; else return value
- `sans_option_unwrap_or(opt:I, default:I) I` — if none, return default; else return value

**Language integration:**
- `some(v)` expression → `sans_option_some(v)` — typeck infers `Option<typeof(v)>`
- `none` literal → `sans_option_none()` — typeck requires context to infer inner type (assignment target, return type, or explicit annotation)
- Pattern matching: `match opt { some(x) => ..., none => ... }`
- `?` operator on Option: `opt?` returns inner value or propagates none to caller (caller must return Option). **Dual-type `?` handling:** Typeck already knows whether the expression type is `Result<T>` or `Option<T>`. The IR lowers `?` differently based on type: for Result (32 bytes), check tag at offset 0, if err propagate full 32-byte error; for Option (16 bytes), check tag at offset 0, if none propagate a 16-byte none. Codegen emits different branch/return sequences for each. The `?` caller return type must match (Result caller for Result `?`, Option caller for Option `?`).
- `!` operator on Option: `opt!` unwraps, panics on none

**Breaking changes to existing APIs:**
- `Map.get(key)` returns `Option<V>` instead of raw value (0 for missing)
- `Array.find(fn)` returns `Option<T>` instead of raw value
- These are intentional breaking changes — the whole point of Option is to force handling

**Scope GC integration:**
- Track Option allocations with new tag `SCOPE_TAG_OPTION = 6` in rc.sans
- `scope_should_keep` must check if kept value is an Option containing a pointer
- When freeing an Option during scope_exit, inspect inner value: if the Option is `some` and the inner value is a heap-allocated type (String, Array, Map, etc.), the inner value must also be freed. This mirrors the same problem Result has with heap-allocated ok values — both share the "container holding a pointer" pattern. Full fix is in v0.7.2 scope GC hardening; in 0.7.1, Option follows the same (incomplete) behavior as Result.

### IR Instructions
- `IR_OPTION_SOME`, `IR_OPTION_NONE`, `IR_OPTION_IS_SOME`, `IR_OPTION_IS_NONE`, `IR_OPTION_UNWRAP`, `IR_OPTION_UNWRAP_OR`
- Pattern matching instructions reuse existing match infrastructure

### Files Changed
- `compiler/constants.sans` — TY_OPTION, IR_OPTION_*, SCOPE_TAG_OPTION
- `compiler/typeck.sans` — Option type resolution, some/none expressions, ? on Option, method type checking
- `compiler/ir.sans` — IR lowering for Option operations
- `compiler/codegen.sans` — codegen for Option IR instructions
- `runtime/option.sans` — new runtime module
- `runtime/rc.sans` — SCOPE_TAG_OPTION tracking
- `compiler/typeck.sans` — update Map.get() and Array.find() return types

### Test Fixtures
- `option_basic.sans` — some/none construction, is_some/is_none
- `option_unwrap.sans` — unwrap, unwrap_or
- `option_match.sans` — pattern matching on Option
- `option_try.sans` — ? operator propagation
- `option_map_get.sans` — Map.get() returning Option

## 2. Map<K,V> Generalization

### Problem
Map is non-generic: string keys and i64 values only. `Map.get("x")` returns an int. Cannot store strings, structs, or other maps as values. Cannot use int keys.

### Design

**Type representation:**
- Extend the existing type struct. Currently Map has no inner type. Change to store key type and value type:
  - offset 8: key type pointer
  - offset 16: value type pointer
- Short syntax: `M<S I>` for `Map<String, Int>`, `M<I S>` for `Map<Int, String>`

**Runtime changes (runtime/map.sans):**
The current runtime stores values as i64 and keys as string pointers. Since all Sans values are already i64 at the register level (pointers, ints, floats all fit in i64), the runtime can remain mostly unchanged. The key change is:
- **Key hashing:** Currently only hashes strings. For integer keys, hash the integer directly. Allowed key types: `I` and `S` only. **Float keys (`F`) are disallowed** — floating point equality is unreliable (NaN != NaN, rounding errors) making them unsuitable as hash keys. Struct keys are also deferred.
- **Value storage:** Already stores i64 — works for all types. No runtime change needed.

**Typeck changes:**
- `Map.get(key:K) Option<V>` — returns Option (depends on section 1)
- `Map.set(key:K, value:V) I` — type-check key and value against Map's K,V
- `Map.has(key:K) B` — type-check key against K
- `Map.keys() [K]` — return Array<K>
- `Map.vals() [V]` — return Array<V>
- `Map.delete(key:K) I`
- `Map.entries() [(K V)]` — return Array<Tuple<K,V>>

**Backward compatibility:**
- `M` without type params defaults to `M<S I>` (current behavior) for gradual migration
- Explicit `M<S I>` is canonical; `sans fmt` should normalize bare `M` to `M<S I>`

### Files Changed
- `compiler/typeck.sans` — Map type parsing with K,V params, method type checking
- `compiler/ir.sans` — Map IR lowering (mostly unchanged, values already i64)
- `runtime/map.sans` — integer key hashing support
- `compiler/constants.sans` — no new constants needed

### Test Fixtures
- `map_generic.sans` — M<S S>, M<I S>, M<I I> basic operations
- `map_option_get.sans` — get() returning Option<V>
- `map_nested.sans` — M<S M<S I>> (map of maps)

## 3. Distinct Types for Handle Safety

### Problem
`is_i64_compat` in typeck.sans makes ALL opaque types (HttpServer, HttpResponse, HttpRequest, Mutex, Sender, Receiver, JoinHandle, JsonValue) interchangeable with plain integers and with each other. You can pass an SSL handle where a file descriptor is expected.

### Design

**Remove opaque types from `is_i64_compat`:**
```
is_i64_compat(t:I) I {
  tag = type_tag(t)
  if tag == TY_INT || tag == TY_FLOAT || tag == TY_STRING || tag == TY_BOOL ||
     tag == TY_ARRAY || tag == TY_MAP || tag == TY_FN { return 1 }
  0
}
```

Remove: `TY_JSON`, `TY_HTTP_RESP`, `TY_HTTP_SERVER`, `TY_HTTP_REQ`, `TY_JOIN_HANDLE`, `TY_SENDER`, `TY_RECEIVER`, `TY_MUTEX`

**Impact:** These types can no longer be assigned to `I` variables or passed where `I` is expected. They're still i64 at the LLVM level (codegen unchanged), but typeck enforces type safety.

**Audit required:** Find all places where opaque values are currently passed as I or compared with I. Fix each call site to use the correct type. The compiler itself and runtime may need type annotation updates.

### Files Changed
- `compiler/typeck.sans` — modify `is_i64_compat`, audit type checking for opaque types
- Possibly `compiler/ir.sans` — ensure IR lowering handles typed opaque values correctly

### Test Fixtures
- `distinct_type_safety.sans` — verify HttpServer cannot be assigned to I, etc.
- `distinct_type_methods.sans` — verify methods still work on opaque types

## 4. Result Combinators

### Problem
Result handling requires unwrapping into a variable at every step. No way to chain operations on Results without intermediate variables.

### Design

**New methods on Result<T>:**
- `r.map(fn)` → `Result<U>` — if ok, apply fn to value, rewrap as ok. If err, propagate err.
- `r.and_then(fn)` → `Result<U>` — if ok, apply fn (which returns Result<U>). If err, propagate err.
- `r.map_err(fn)` → `Result<T>` — if err, apply fn to error message. If ok, propagate ok.
- `r.or_else(fn)` → `Result<T>` — if err, apply fn (which returns Result<T>). If ok, propagate ok.

**Implementation approach:**
Each combinator is a method on Result that takes a closure argument. **These cannot be implemented as pure runtime functions** because Sans has no `sans_closure_call` runtime function — closures are dispatched through codegen-emitted indirect calls (`emit_closure_aware_call` in codegen.sans). Combinators must follow the same pattern as `Array.map`/`Array.filter`: IR instructions that codegen lowers to inline closure dispatch.

**Typeck:** Check that fn's parameter type matches Result's inner T (for map/and_then) or S (for map_err). Infer return type from fn's return type.

**IR + Codegen:** New instructions `IR_RESULT_MAP`, `IR_RESULT_AND_THEN`, `IR_RESULT_MAP_ERR`, `IR_RESULT_OR_ELSE`. Codegen for each:
1. Load result tag (offset 0)
2. Branch: if ok (tag == 0), load value (offset 8), call closure via `emit_closure_aware_call1(fn, value)`, wrap return in new Result
3. If err, propagate the original result unchanged
4. `and_then` differs: the closure returns a Result directly, no re-wrapping needed

### Files Changed
- `compiler/typeck.sans` — add map/and_then/map_err/or_else method types
- `compiler/ir.sans` — IR lowering for combinator calls
- `compiler/codegen.sans` — codegen for combinator IR instructions
- `runtime/result.sans` — runtime implementations

### Test Fixtures
- `result_map.sans` — chaining map on ok/err results
- `result_and_then.sans` — chaining and_then
- `result_combinators.sans` — complex chains

## 5. Trait Objects / Dynamic Dispatch

### Problem
Traits exist but are checked statically only — no `dyn Trait` or polymorphic collections. Cannot have `[dyn Printable]` (array of different types that all implement Printable).

### Design

**Trait object representation — fat pointer (16 bytes):**
```
offset 0: data pointer (ptr to actual value, type-erased)
offset 8: vtable pointer (ptr to method table for that concrete type)
```

**Vtable layout:**
Array of function pointers, one per trait method, in declaration order.
```
vtable for Printable impl on Point:
  [0]: ptr to Point_print
  [1]: ptr to Point_to_string
  ...
```

**Syntax:**
- `dyn Trait` as a type — represents any value implementing Trait
- `x as dyn Trait` — explicit coercion from concrete type to trait object
- Method calls on `dyn Trait` dispatch through vtable

**Type system:**
- New type tag: `TY_DYN_TRAIT` with inner pointing to trait definition
- **Storage model:** Heap-allocate the fat pointer as a 16-byte struct (data ptr + vtable ptr), return a single i64 pointer to it. This is consistent with how Result (32 bytes) and Option (16 bytes) work — they're heap-allocated and passed as i64 pointers. This avoids breaking the single-i64 register model that the entire compiler assumes.
- Trait objects ARE i64_compat (they're a pointer to a 16-byte heap struct, like Result/Option)
- Track with `SCOPE_TAG_DYN_TRAIT = 7` in rc.sans for scope GC

**Limitations in 0.7.1:**
- No trait inheritance (trait A: B)
- No default implementations
- No associated types
- Trait objects cannot be used in generic contexts (no `T: Trait` bounds)
- These are deferred to post-1.0

**Codegen:**
- When creating trait object: emit vtable as global constant, alloc 16 bytes, store data ptr at offset 0 and vtable ptr at offset 8, return pointer
- When calling method: load data ptr (offset 0), load vtable ptr (offset 8), index into vtable, call through function pointer with data ptr as first argument
- vtables generated per (concrete type, trait) pair. **Data flow from typeck to codegen:** typeck builds a `trait_impls` map keyed by `"ConcreteType::TraitName"` → array of (method_name, mangled_fn_name) pairs. This map is passed through IR context to codegen, which emits the vtable globals referencing the concrete method function pointers.

### Files Changed
- `compiler/constants.sans` — TY_DYN_TRAIT
- `compiler/typeck.sans` — dyn type resolution, coercion checking, method dispatch through traits
- `compiler/ir.sans` — IR lowering for vtable construction and dispatch
- `compiler/codegen.sans` — vtable global emission, indirect calls through vtable

### Test Fixtures
- `trait_object_basic.sans` — create trait object, call method
- `trait_object_array.sans` — array of dyn Trait, iterate and call methods
- `trait_object_coercion.sans` — explicit `as dyn Trait` coercion

## Implementation Order

1. **Option<T>** — foundational, Map.get() depends on it
2. **Map<K,V> generalization** — depends on Option
3. **Distinct types** — independent, small change, high safety value
4. **Result combinators** — independent, uses existing closure infra from 0.7.0
5. **Trait objects** — most complex, build last

Items 3 and 4 can be done in parallel with 1+2.

---

## Deferred from v0.7.1

- **Trait objects / dynamic dispatch (`dyn Trait`)** — deferred to v0.7.2. Too complex and risky for this release (vtable emission, indirect dispatch, new type coercion). The foundation (traits, impl blocks) exists but dynamic dispatch requires significant codegen work.
- **Array.find() returning Option<T>** — deferred. Map.get() was migrated to Option; Array.find() can follow in a later release.
