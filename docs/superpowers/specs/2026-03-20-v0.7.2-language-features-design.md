## 1. `defer` Statement

### Problem
No way to guarantee resource cleanup (file handles, sockets, locks). Scope GC handles memory, but non-memory resources leak on early return, break, or panic.

### Design

**Syntax:** `defer expr` — executes `expr` when the enclosing function returns, in LIFO order.
```
f = fopen("data.txt")
defer fclose(f)
// ... work with f ...
// fclose(f) runs automatically on any return path
```

**Semantics (Go-compatible):**
- Deferred expressions are evaluated in LIFO order (last defer runs first)
- Arguments to deferred calls are evaluated at the defer site, not at exit
- Defer runs when the enclosing function returns (normal return or early return). Loop control flow (`break`/`continue`) does NOT trigger defer — only function exit does.
- Defer does NOT run on `exit()` — that terminates the process immediately
- Each function has its own defer stack. Closures containing `defer` manage an independent defer stack from their enclosing function.

**Implementation — defer stack:**

**Parser:**
- New token `TK_DEFER`, new statement type `ST_DEFER`
- `ST_DEFER` stores: `[tag:I, size:I, expr:I, line:I, col:I]`

**IR lowering:**
- New instruction `IR_DEFER_PUSH` — pushes a deferred expression onto the function's defer stack
- At every `IR_RET` site, emit `IR_DEFER_RUN_ALL` before the return
- The defer stack is a per-function array of (fn_ptr, args...) entries

**Codegen:**
- Defer stack is a local alloca'd array (max 32 defers per function, compile error if exceeded)
- `IR_DEFER_PUSH`: store fn_ptr + captured args into next stack slot, increment counter
- `IR_DEFER_RUN_ALL`: loop from counter-1 to 0, call each deferred function with its saved args
- Insert `defer_run_all` before every `ret` instruction and before `scope_exit`

**Interaction with scope GC:**
- Defer runs BEFORE scope_exit (deferred closures may reference scope-tracked memory)
- Order: defer_run_all → scope_exit → ret

### Files Changed
- `compiler/constants.sans` — TK_DEFER, ST_DEFER, IR_DEFER_PUSH, IR_DEFER_RUN_ALL
- `compiler/lexer.sans` — recognize `defer` keyword
- `compiler/parser.sans` — parse defer statement
- `compiler/typeck.sans` — type-check deferred expression
- `compiler/ir.sans` — IR lowering with defer stack management
- `compiler/codegen.sans` — defer stack alloca, push, run-all emission at ret sites

## 2. Channel `select`

### Problem
Cannot multiplex channels. Cannot implement timeouts, cancellation, or fan-in patterns. Current channels only support blocking send/recv on a single channel.

### Design

**Syntax:**
```
select {
  msg = ch1.recv() => handle_msg(msg)
  ch2.recv() => handle_other()
  timeout(1000) => p("timed out")
}
```

**Semantics:**
- Blocks until one of the cases is ready
- If multiple are ready, picks the first ready in declaration order (deliberate divergence from Go's random selection — deterministic behavior is more predictable for AI-generated code and simpler to implement)
- **Known limitation:** The declaration-order priority creates starvation risk — a high-throughput channel as case 1 can starve later cases. Users should place cancellation/timeout cases early. This will be addressed in v0.11 (performance milestone) if needed.
- `timeout(ms)` case fires after ms milliseconds with no other case ready
- Each case can be a `.recv()` (receive from channel) or `.send(val)` (send to channel)
- The received value is bound to a variable if `var = ch.recv()` form is used
- `select {}` with no cases blocks forever (consistent with Go). The polling implementation uses an infinite sleep loop.

**Implementation:**

**Parser:**
- New token `TK_SELECT`
- Select parsed as a special statement with N cases
- Each case: `[channel_expr, operation (recv/send), optional_bind_var, body_stmts]`

**IR lowering:**
- `IR_SELECT_BEGIN` — initialize select state
- `IR_SELECT_CASE_RECV(channel, bind_var)` — register a recv case
- `IR_SELECT_CASE_SEND(channel, value)` — register a send case
- `IR_SELECT_CASE_TIMEOUT(ms)` — register timeout case
- `IR_SELECT_WAIT` — block until a case fires, returns case index
- Branch on case index to execute the corresponding body

**Codegen — polling approach:**
Since channels use pthread mutexes/condvars, implement select as:
1. Try each channel non-blockingly (trylock + check count)
2. If one is ready, execute that case
3. If none ready, sleep briefly and retry (with exponential backoff up to 1ms)
4. Check timeout between retries
5. This is simpler than a full condvar-based select but less efficient — acceptable for 0.7.x

**Alternative (better, more complex):** Use a shared condvar that all channels signal into. Each channel case registers a waiter. When any channel becomes ready, it signals the shared condvar. This avoids polling but requires modifying the channel runtime.

**Recommendation:** Start with polling approach. Upgrade to condvar-based in v0.11 (performance milestone) if needed.

### Files Changed
- `compiler/constants.sans` — TK_SELECT, ST_SELECT, IR_SELECT_*
- `compiler/lexer.sans` — recognize `select` keyword
- `compiler/parser.sans` — parse select statement with cases
- `compiler/typeck.sans` — type-check channel operations in cases
- `compiler/ir.sans` — IR lowering for select
- `compiler/codegen.sans` — polling-based select implementation

### Test Fixtures
- `select_basic.sans` — two channels, receive from whichever is ready first
- `select_timeout.sans` — timeout case fires when no channel ready
- `select_send.sans` — select with send cases
- `select_multi.sans` — fan-in pattern with multiple producers

## 3. Module System Improvements

### Problem
Current module system is flat: `import "module"` makes everything available as `module.func()`. No namespaced imports, no visibility control, no re-exports.

### Design

**3a. Namespaced imports:**
```
import "http" as h       // alias: h.get(), h.post()
import "json" { parse stringify }  // selective: parse(), stringify() (no prefix)
```

**Parser:** Extend import parsing to handle `as` alias and `{ ... }` selective list.

**Resolution:** In main.sans import resolution, store the alias/selection info alongside the resolved module. In typeck, use alias for qualified lookups.

**3b. Visibility modifiers:**
```
pub greet(name:S) S = "Hello " + name    // exported
helper(x:I) I = x * 2                    // private (default)
```

**Default: private.** Only `pub` functions/structs/enums are visible outside the module.

**Parser:** Recognize `pub` keyword before function/struct/enum declarations. Store visibility flag on the declaration AST node.

**Typeck:** When checking cross-module references, verify the target has `pub` visibility. Emit error if accessing a private member.

**3c. Re-exports:**
```
pub import "utils" { helper }  // re-exports helper from utils
```

Deferred to later if complexity is too high. Namespaced imports + visibility are the priority.

### Files Changed
- `compiler/constants.sans` — TK_PUB, TK_AS (if not already present)
- `compiler/lexer.sans` — recognize `pub`, `as` keywords
- `compiler/parser.sans` — extended import syntax, pub modifier on declarations
- `compiler/typeck.sans` — visibility checking on cross-module access
- `compiler/main.sans` — alias/selective import in module resolution

**Backward compatibility:** When `pub` enforcement first lands, all existing cross-module function calls will break unless those functions are marked `pub`. Migration approach: in 0.7.2, emit a **warning** (not error) for cross-module access to non-pub functions. The warning becomes an error in 0.8.0. This gives existing code and examples time to add `pub` annotations.

### Test Fixtures
- `import_alias/` — import with `as` alias
- `import_selective/` — selective imports
- `visibility_pub/` — pub vs private access
- `visibility_warning/` — non-pub cross-module access emits warning

## 4. Dead Code Elimination

### Problem
All functions and imports are compiled even if unused. Binary size grows, compile time wasted.

### Design

**Phase 1 — Function-level DCE:**
1. Build a call graph during typeck or IR lowering (which functions call which)
2. Mark `main` as live
3. Walk call graph transitively — mark all reachable functions as live
4. Skip IR generation and codegen for unreachable functions

**Phase 2 — Constant folding:**
- Evaluate `2 + 3` → `5` at compile time in IR
- Fold `if true { a } else { b }` → `a`
- Handle integer arithmetic, boolean logic, string concatenation of literals

**Implementation location:** New pass between typeck and IR lowering, or as part of IR lowering.

### Files Changed
- `compiler/ir.sans` — call graph construction, reachability marking, constant folding
- `compiler/codegen.sans` — skip dead functions

### Test Fixtures
- `dce_unused_fn.sans` — unused function not in binary (verify by checking output size or absence of symbol)
- `const_fold.sans` — constant expressions evaluated at compile time

## 5. Scope GC Hardening

### Problem
Known gaps in scope GC cause use-after-free or memory leaks:
- Closures capturing freed variables
- Nested loops accumulating allocations
- Depth 2+ containers not preserved on return
- Cross-function promotion edge cases

### Design

**5a. Closure capture promotion:**
When a closure captures a variable, and that variable is scope-tracked in the enclosing function, the variable must be promoted out of the enclosing function's scope so it survives scope_exit.

Implementation: In codegen, when emitting closure creation, for each captured variable that is scope-tracked, call `sans_scope_promote(ptr)` which moves it from the current scope to the parent scope (or removes it from scope tracking entirely — the closure now owns it).

Add `sans_scope_promote(ptr:I)` to `runtime/rc.sans`: walk the alloc list, find the node for `ptr`, remove it from the current scope's range.

**5b. Recursive scope_should_keep:**
Currently `scope_should_keep` in `rc.sans` only checks direct array elements (depth 1). Fix to recurse:
1. If kept value is an array: walk elements, keep each, recurse into elements that are arrays/maps/results
2. If kept value is a map: walk values, keep each, recurse
3. If kept value is a result (ok): keep the inner value, recurse
4. **Cycle detection mechanism:** Add a `visited` flag (1 byte) at offset 24 of each tracking node, expanding nodes from 24 to 32 bytes (with alignment). Before recursing into a node, check its visited flag; if set, skip. Reset all visited flags to 0 during the scope_exit walk before starting the keep-check. This avoids needing a separate hash set (which would itself depend on scope GC — a chicken-and-egg problem) and is O(1) per node.
5. Use the allocation tag to determine what to recurse into

**5c. Loop allocation cleanup:**
For `while` loops, emit `scope_exit` at the end of each iteration to free per-iteration allocations. This requires a per-iteration scope: `scope_enter` at loop top, `scope_exit(0)` at loop bottom (keep nothing).

Careful: loop variables that accumulate across iterations (e.g., appending to an array) must be excluded from per-iteration cleanup. The approach: only free allocations made within the current iteration, not those from outer scope.

**5d. Document self-compilation patterns:**
Create a document listing the specific patterns in the compiler's own code that break under scope GC, with minimal reproducing examples. This informs future fixes without blocking the release.

### Files Changed
- `runtime/rc.sans` — recursive scope_should_keep, scope_promote, per-iteration cleanup
- `compiler/codegen.sans` — closure capture promotion, loop scope enter/exit emission

### Test Fixtures
- `scope_gc_closure.sans` — closure capturing a heap-allocated value survives scope_exit
- `scope_gc_nested_return.sans` — returning [["a" "b"] ["c" "d"]] preserves inner strings
- `scope_gc_loop.sans` — while loop doesn't leak per-iteration allocations
- `scope_gc_map_return.sans` — returning a map preserves all values

## 6. Runtime Hash Table Deduplication

### Problem
`runtime/map.sans` and `runtime/json.sans` contain identical DJB2 hash + linear probing implementations. Bug fixes need to land in two places.

### Design

Extract shared hash table to `runtime/hashtable.sans`:
- `sans_ht_hash(key:I) I` — DJB2 hash
- `sans_ht_str_eq(a:I, b:I) I` — string comparison
- `sans_ht_probe(hashes:I, keys:I, cap:I, hash:I, key:I) I` — linear probing

Both `map.sans` and `json.sans` import and call these shared functions.

### Files Changed
- `runtime/hashtable.sans` — new shared module
- `runtime/map.sans` — replace inline hash/probe with calls to hashtable
- `runtime/json.sans` — replace inline hash/probe with calls to hashtable

### Test Fixtures
- Existing map and JSON tests validate no regressions

## Implementation Order

1. **Defer** — independent, straightforward
2. **Module improvements** — independent, useful for later testing
3. **Dead code elimination** — independent
4. **Scope GC hardening** — independent but test-intensive
5. **Runtime deduplication** — independent, low risk
6. **Channel select** — most complex, build last

All items are largely independent. 1-3 can be parallelized. 4-5 can be parallelized.
