# v0.7.2 Language Features & Compiler Hygiene — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add defer statement, fix remaining v0.7.1 gaps (Array.find→Option, curl bootstrap), add module visibility, harden scope GC, deduplicate runtime, and add dead code elimination.

**Architecture:** The Sans compiler pipeline is: lexer → parser → typeck → IR → codegen → LLVM. New keywords (defer, pub) require changes at every stage. Compiler hygiene items (scope GC, runtime dedup, DCE) are more targeted. Trait objects and channel select are deferred to v0.7.3 due to complexity.

**Tech Stack:** Sans (self-hosted compiler), LLVM 17 (backend)

**Spec:** `docs/superpowers/specs/2026-03-20-v0.7.2-language-features-design.md`

**Build steps:** `cp /tmp/sans ~/.local/bin/sans && sans build compiler/main.sans && cp compiler/main ~/.local/bin/sans`
If `/tmp/sans` doesn't exist, download: `curl -fsSL "https://github.com/sans-language/sans/releases/download/v0.7.0/sans-macos-arm64.tar.gz" | tar xz -C /tmp/ && install_name_tool -change /opt/homebrew/opt/openssl@3/lib/libssl.3.dylib $(brew --prefix openssl@3)/lib/libssl.3.dylib /tmp/sans && install_name_tool -change /opt/homebrew/opt/openssl@3/lib/libcrypto.3.dylib $(brew --prefix openssl@3)/lib/libcrypto.3.dylib /tmp/sans`

## Scope Decision

The v0.7.2 spec includes 9 items. Based on complexity and risk, this plan splits them:

**In scope (this plan):**
1. Array.find() → Option<T> (quick fix, leftover from v0.7.1)
2. Curl runtime bootstrap fix (investigation + fix)
3. `defer` statement (medium — new keyword, parser, IR, codegen)
4. `pub` visibility modifier (medium — parser, typeck)
5. Runtime hash table deduplication (small — extract shared code)
6. Scope GC hardening — recursive scope_should_keep (medium — runtime)

**Deferred to v0.7.3:**
- Channel `select` (complex — polling/condvar, multiple channel cases)
- Trait objects / `dyn Trait` (complex — vtable emission, indirect dispatch)
- Dead code elimination (already implemented at basic level; interprocedural DCE is large)
- Namespaced imports (`import "x" as y`) and re-exports

```sans
sans_hash_string(key:I) I {
  h := 5381
  i := 0
  c := load8(key)
  while c != 0 {
    h = h * 33 + c
    i = i + 1
    c = load8(key + i)
    0
  }
  if h == 0 { 1 } else { h }
}

sans_str_eq(a:I b:I) I {
  i := 0
  while 1 == 1 {
    ca = load8(a + i)
    cb = load8(b + i)
    if ca != cb { return 0 }
    if ca == 0 { return 1 }
    i = i + 1
    0
  }
  0
}
```

**Files:**

```sans
}

  s = result[0]
  slen(s)  // "hello" = 5
}
```

- [ ] **Step 2: Add visited flag to tracking nodes**

In `runtime/rc.sans`, tracking nodes are currently 24 bytes: `[ptr:I, tag:I, next:I]`. Expand to 32 bytes with a visited flag at offset 24: `[ptr:I, tag:I, next:I, visited:I]`.

Update `sans_scope_track` to allocate 32 bytes instead of 24. Initialize visited=0 at offset 24.

- [ ] **Step 3: Implement recursive scope_should_keep**

Replace the current depth-1-only implementation with a recursive walker:

```sans
scope_should_keep(ptr:I keep:I) I {
  if ptr == keep { return 1 }
  if keep == 0 { return 0 }
  // Find keep's tag in tracking list
  tn := rc_alloc_head
  keep_tag := -1
  while tn != 0 {
    if load64(tn) == keep { keep_tag = load64(tn + 8) }
    tn = load64(tn + 16)
  }
  // Array: check elements and recurse
  if keep_tag == 1 {
    arr_data = load64(keep)
    arr_len = load64(keep + 8)
    ei := 0
    while ei < arr_len {
      elem = load64(arr_data + ei * 8)
      if elem == ptr { return 1 }
      // Recurse into element if it's tracked
      if scope_should_keep(ptr, elem) == 1 { return 1 }
      ei += 1
      0
    }
  }
  // Map: check values and recurse
  if keep_tag == 2 {
    vals = load64(keep + 16)
    cap = load64(keep + 32)
    hashes = load64(keep)
    mi := 0
    while mi < cap {
      h = load64(hashes + mi * 8)
      if h != 0 {
        val = load64(vals + mi * 8)
        if val == ptr { return 1 }
        if scope_should_keep(ptr, val) == 1 { return 1 }
      }
      mi += 1
      0
    }
  }
  // Result: check value (offset 8)
  if keep_tag == 4 {
    val = load64(keep + 8)
    if val == ptr { return 1 }
    if scope_should_keep(ptr, val) == 1 { return 1 }
  }
  // Option: check value (offset 8) if some (tag==1)
  if keep_tag == 6 {
    tag = load64(keep)
    if tag == 1 {
      val = load64(keep + 8)
      if val == ptr { return 1 }
      if scope_should_keep(ptr, val) == 1 { return 1 }
    }
  }
  0
}
```

**NOTE:** This recursive version has no cycle detection yet. For v0.7.2, cycles are unlikely in practice (Sans doesn't have circular data structures). Add cycle detection (visited flag) if needed.

```bash

```

### Task 7: defer Codegen

**Files:**
- Modify: `compiler/codegen.sans`
- Create: `tests/fixtures/defer_basic.sans`
- Create: `tests/fixtures/defer_lifo.sans`

- [ ] **Step 1: Implement defer stack in codegen**

At function entry, allocate a defer stack (alloca array of i64 pairs: fn_ptr + arg). Track a defer counter.

`IR_DEFER_PUSH`: store fn_ptr and arg into the next stack slot, increment counter.

`IR_DEFER_RUN_ALL`: loop from counter-1 to 0, load fn_ptr and arg from each slot, call fn_ptr(arg).

- [ ] **Step 2: Create test fixtures**

`tests/fixtures/defer_basic.sans`:
```sans

  cleanup_ran = 1
  0
}

main() I {
  defer do_cleanup()
  cleanup_ran
}
```
Expected: This tests that defer runs after the function body but the test checks before return... Actually defer runs at function EXIT. Need a different test approach — use a global or return value.

Better test:
```sans
main() I {
  a := 0
  // defer should run AFTER the function body evaluates
  // but BEFORE the return value is used
  // Actually: defer runs when the FUNCTION returns, so we need
  // a helper function to observe the effect
  0
}
```

This is tricky to test with exit codes. Use a simpler approach:
```sans

  defer p("deferred\n")

```
Expected exit: 42 (and "deferred" printed to stdout, but we can't check stdout in exit-code tests)
