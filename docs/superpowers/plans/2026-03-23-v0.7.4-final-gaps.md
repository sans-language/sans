## Task 2: Loop-Scoped Allocation Cleanup

**Goal:** Prevent unbounded memory growth in long-running loops by adding per-iteration scope cleanup.

**Design:** Emit `scope_enter()`/`scope_exit()` around while-loop bodies. The scope_exit at loop iteration end frees allocations from that iteration. Values that escape the loop (assigned to variables declared before the loop) must be promoted, same as function-return promotion.

This is the most impactful correctness fix — without it, server handler loops and any `while true` pattern leak memory until function return.

**Files:**
- Modify: `compiler/ir.sans:1558-1584` (ST_WHILE lowering — add scope_enter/scope_exit IR instructions around loop body)
- Modify: `compiler/codegen.sans` (emit scope_enter/scope_exit calls from new IR instructions)
- Modify: `runtime/rc.sans` (may need nested scope support — scope_enter inside scope_enter)
- Create: `tests/fixtures/loop_scope_basic.sans` (allocations in loop freed per iteration)
- Create: `tests/fixtures/loop_scope_escape.sans` (value escapes loop — not freed)
- Create: `tests/fixtures/loop_scope_nested.sans` (nested loops with allocations)

- [ ] **Step 5: Investigate the escape problem**

**This must be resolved before writing any implementation code.**

The core difficulty: values assigned to variables declared *before* the loop must not be freed by the loop's scope_exit. `scope_exit(return_val)` promotes only one value. For loops, we need to understand how outer-variable assignment interacts with scope tracking.

Investigation steps:
1. Read `runtime/rc.sans` — does `sans_rc_track(ptr)` add to the *current* scope's allocation list? If so, a string created inside a loop and assigned to an outer variable would be tracked in the loop scope and freed by `scope_exit`.
2. Read `compiler/codegen.sans` — when a new string/array/map is created and assigned to an existing variable, is `sans_rc_track` called? Or is tracking only done at allocation time?
3. Write a minimal test: loop that creates a string, assigns it to an outer variable, then uses it after the loop. If the value is freed by scope_exit, this will segfault.

**Decision point:** Based on findings:
- If outer-variable stores don't go through scope tracking → `scope_exit(0)` is safe. Proceed to Step 6.
- If they do → we need a `scope_keep(ptr)` API or a different approach (e.g., only emit loop scope for loops with no outer assignments, or promote all outer-assigned values before scope_exit). Design the solution and document it before proceeding.

- [ ] **Step 6: Implement loop scope emission**

- [ ] **Step 5: Implement parser — struct and tuple pattern recognition**

In `compiler/parser.sans` `parse_pattern()`:

**Struct patterns** — add after the enum variant check (line ~1738). When we see an uppercase identifier NOT followed by `::`, check if followed by `{`:

```
// Struct destructuring pattern: Name { field1, field2 }
if p_peek_kind_at(p, 1) == TK_LBRACE {
  struct_name = p_expect_ident(p)
  p_expect(p, TK_LBRACE)
  fields = array<I>()
  while p_at(p, TK_RBRACE) == 0 {
    field = p_expect_ident(p)
    fields.push(field)
    if p_at(p, TK_COMMA) == 1 { p_advance(p) }
  }
  p_expect(p, TK_RBRACE)
  pat = make_pattern(struct_name, 0, fields)
  store64(pat + 24, 5)  // pat_type = 5 (struct)
  return pat
}
```

**Tuple patterns** — add at the top of `parse_pattern()`, before integer/string checks. When we see `(`:

```
// Tuple destructuring pattern: (a, b, c)
if kind == TK_LPAREN {
  p_advance(p)
  bindings = array<I>()
  while p_at(p, TK_RPAREN) == 0 {
    bindings.push(parse_pattern(p))  // recursive — allows nested patterns
    if p_at(p, TK_COMMA) == 1 { p_advance(p) }
  }
  p_expect(p, TK_RPAREN)
  pat = make_pattern(0, 0, bindings)
  store64(pat + 24, 6)  // pat_type = 6 (tuple)
  return pat
}
```

Reference: tuple destructuring in `let` statements already exists at `parser.sans:1063` — follow the same element extraction approach.

- [ ] **Step 6: Implement typeck — struct and tuple pattern validation**

In `compiler/typeck.sans`, where match patterns are type-checked (around line 3104-3194):

**pat_type 5 (struct):**
- Look up the struct by name in the structs map
- Validate each field name exists in the struct definition
- Bind each field name to the field's type in the match arm's local scope

**pat_type 6 (tuple):**
- Verify the scrutinee is a tuple type
- Verify the number of bindings matches the tuple arity
- Bind each element to the corresponding tuple element's type

- [ ] **Step 7: Implement codegen — struct and tuple field extraction**

In `compiler/codegen.sans`, where match patterns are compiled:

**pat_type 5 (struct):** For each field binding, emit a load from the struct pointer at the field's offset. Store into the binding's register.

**pat_type 6 (tuple):** For each binding, emit a load from the tuple pointer at the element's offset (element_index * 8). Store into the binding's register. Reference the existing tuple element extraction in codegen for `let (a, b) = ...` statements.

## Task 4: Generic Monomorphization Depth Guard

**Goal:** Prevent infinite recursion or exponential blowup in generic instantiation by adding a depth limit.

**Design:** Add a recursion depth counter to `instantiate_generic_struct()` in typeck.sans. If depth exceeds a limit (e.g., 32), emit a compile error. This is a safety guard — no known programs hit this, but without it, a typo in a recursive generic type can hang the compiler.

**Files:**
- Modify: `compiler/typeck.sans:934-1038` (`instantiate_generic_struct` — add depth parameter/global)
- Create: `tests/fixtures/generic_deep_nesting.sans` (3-4 levels of nested generics — should compile)
- Create: `tests/negative/generic_too_deep.sans` (pathological nesting — should error)

### Steps

- [ ] **Step 1: Write test — valid deep nesting**

- [ ] **Step 2: Write negative test — excessive depth**

The negative test needs to trigger 33+ levels of recursive monomorphization. Writing a 33-deep literal is impractical. Instead, use a programmatic approach — generate the test file with a script:

```bash
# Generate tests/negative/generic_too_deep.sans
echo 'struct W<T> { v:T }' > tests/negative/generic_too_deep.sans
echo 'main() I {' >> tests/negative/generic_too_deep.sans

TYPE="I"

echo '}' >> tests/negative/generic_too_deep.sans
```

This generates a valid Sans file with `W<W<W<...I...>>>` nested 35 deep. The value construction (`{ v: 0 }`) won't be correct for all 35 levels, but monomorphization of the *type* should trigger the depth guard during struct instantiation before codegen reaches value construction.

**Alternative:** If the above doesn't trigger monomorphization (because the type is invalid before reaching depth 35), add a simpler guard: track total monomorphization count per compilation and error if it exceeds 1000. This catches exponential blowup without requiring deep nesting in a single type.
