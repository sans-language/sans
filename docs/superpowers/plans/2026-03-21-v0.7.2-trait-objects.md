# v0.7.2 Trait Objects + Cleanup — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement trait objects (dyn Trait) with vtable dispatch and polymorphic collections. Fix Array.find→Option and curl bootstrap.

**Architecture:** Trait infrastructure already exists (trait definitions, impl blocks, method dispatch via mangled names). Trait objects add: a new type (TY_DYN_TRAIT=22), coercion via `as dyn Trait`, vtable globals in LLVM IR, and indirect method dispatch. Fat pointer is heap-allocated 16 bytes (data ptr + vtable ptr), consistent with Result/Option.

**Tech Stack:** Sans (self-hosted compiler), LLVM 17

**Spec:** `docs/superpowers/specs/2026-03-21-v0.7.2-trait-objects-design.md`

**Build steps:** `cp /tmp/sans ~/.local/bin/sans && sans build compiler/main.sans && cp compiler/main ~/.local/bin/sans`

## Chunk 3: Vtable Generation and Method Dispatch

### Task 5: Build trait_impls Map in Typeck

**Files:**
- Modify: `compiler/typeck.sans`

- [ ] **Step 1: Create trait_impls map during impl validation**

During Pass 0d (impl validation, line 3355-3415), for each `impl Trait for Type`:
- Build key: `"TypeName::TraitName"`
- Build value: array of `(method_name, mangled_fn_name)` pairs
- Mangled name follows existing pattern: `TypeName_methodName` (or `module__TypeName__methodName`)
- Store in a global `trait_impls` map

```sans
// At top of typeck (global):
g trait_impls = 0  // initialized in check_module

// During impl processing:
if impl_trait_name != 0 {
  // ... existing validation ...
  // NEW: build impl entry for vtable generation
  impl_entry = array<I>()
  mi := 0
  while mi < impl_methods.len() {
    m = impl_methods.get(mi)
    mname = load64(m)  // method name from impl AST
    mangled = target_type_name + "_" + mname
    pair = alloc(16)
    store64(pair, mname)
    store64(pair + 8, mangled)
    impl_entry.push(pair)
    mi = mi + 1
    0
  }
  impl_key = target_type_name + "::" + impl_trait_name
  mset(trait_impls, impl_key, impl_entry)
}
```

### Task 6: Vtable Emission in Codegen

**Files:**
- Modify: `compiler/ir.sans` (IR lowering for EX_AS_DYN)
- Modify: `compiler/codegen.sans` (vtable globals + IR_DYN_TRAIT_CREATE)

- [ ] **Step 1: IR lowering for EX_AS_DYN**

In ir.sans, in `lower_expr`, add handling for `EX_AS_DYN`:
```sans
if tag == EX_AS_DYN {
  src = load64(expr + 16)
  src_reg = lower_expr(ctx, src)
  dest = ctx_fresh_reg(ctx)
  // Store trait name and source type info for codegen
  // The vtable reference is resolved at codegen time
  ctx_emit(ctx, ir_inst1(IR_DYN_TRAIT_CREATE, dest, src_reg))
  // Attach metadata: trait name + source type name
  // (Need a way to pass this to codegen — either extra IR fields or a side map)
  dest
}
```

**Design decision for metadata:** The codegen needs to know which vtable to use. Options:
- A) Encode vtable name as a string constant IR operand
- B) Use a side map from IR instruction → vtable info
- C) Use two-operand IR with vtable reference as second operand

Recommend A: emit `IR_DYN_TRAIT_CREATE(dest, src_reg, vtable_name_str)` — use `ir_inst2` where the second operand is a string constant register holding the vtable global name.

- [ ] **Step 2: Emit vtable globals in codegen**

In codegen.sans, in the function that emits globals/declarations (before function bodies), emit vtable constants. For each entry in trait_impls:

```llvm
@vtable.Point.Valued = private constant [1 x ptr] [ptr @Point_value]
```

The vtable is an array of function pointers, one per trait method, in trait declaration order.

To do this, codegen needs access to:
- `trait_impls` map (Type::Trait → method/mangled pairs)
- `trait_registry` (Trait → ordered method list, for correct vtable ordering)

Both need to be passed from typeck through IR context to codegen.

- [ ] **Step 3: Compile IR_DYN_TRAIT_CREATE**

Codegen for `IR_DYN_TRAIT_CREATE(dest, src, vtable_name)`:
```llvm
%fat_ptr = call ptr @malloc(i64 16)
%data_as_ptr = inttoptr i64 %src to ptr
store ptr %data_as_ptr, ptr %fat_ptr           ; offset 0: data
%vtable_ptr = getelementptr [N x ptr], ptr @vtable.Type.Trait, i64 0, i64 0
%vt_gep = getelementptr i8, ptr %fat_ptr, i64 8
store ptr %vtable_ptr, ptr %vt_gep             ; offset 8: vtable
%result = ptrtoint ptr %fat_ptr to i64
```

Scope track with tag 7 (SCOPE_TAG_DYN_TRAIT).

In `runtime/rc.sans`, in `scope_free`, add after SCOPE_TAG_OPTION:

### Task 7: Method Dispatch on dyn Trait

**Files:**
- Modify: `compiler/typeck.sans` (method dispatch for TY_DYN_TRAIT)
- Modify: `compiler/ir.sans` (IR lowering for dyn Trait method calls)
- Modify: `compiler/codegen.sans` (IR_DYN_TRAIT_CALL codegen)

- [ ] **Step 1: Type-check method calls on dyn Trait**

In typeck.sans, in the method call handler (EX_METHOD_CALL), add a case for when `obj_type` is `TY_DYN_TRAIT`:

```sans
if ot == TY_DYN_TRAIT {
  trait_name = load64(obj_ty + 8)
  // Look up trait in trait_registry to get method signatures
  trait_methods = mget(trait_registry, trait_name)
  // Find the method by name, get its index and return type
  mi := 0
  while mi < trait_methods.len() {
    tm = trait_methods.get(mi)
    if load64(tm) == mc_method {
      // Found — check args, return the method's return type
      // tm offset 8: param types, tm offset 16: return type
      return load64(tm + 16)
    }
    mi = mi + 1
    0
  }
  return tc_error_at(expr, "trait '" + trait_name + "' has no method '" + mc_method + "'")
}
```

- [ ] **Step 2: IR lowering for dyn Trait method calls**

In ir.sans, when lowering a method call and the object type is IRTY for dyn trait, emit `IR_DYN_TRAIT_CALL`:

```sans
// obj_reg holds the trait object pointer
// method_index is determined from trait method order
// args are the explicit arguments (not self — self is extracted from the fat pointer)
dest = ctx_fresh_reg(ctx)
ctx_emit(ctx, ir_dyn_call(IR_DYN_TRAIT_CALL, dest, obj_reg, method_index, arg_regs))
```

The IR instruction needs: dest, trait_object, method_index, and args. This may need a custom IR encoding since `ir_inst2` only supports 2 operands.

- [ ] **Step 3: Codegen for IR_DYN_TRAIT_CALL**

```llvm
; Load fat pointer
%fat = inttoptr i64 %trait_obj to ptr
; Load data ptr (offset 0)
%data_ptr = load ptr, ptr %fat
%data = ptrtoint ptr %data_ptr to i64
; Load vtable ptr (offset 8)
%vt_gep = getelementptr i8, ptr %fat, i64 8
%vtable = load ptr, ptr %vt_gep
; Index into vtable
%fn_gep = getelementptr ptr, ptr %vtable, i64 METHOD_INDEX
%fn_ptr = load ptr, ptr %fn_gep
; Call through function pointer with data as first arg (self)
%result = call i64 %fn_ptr(i64 %data, i64 %arg1, ...)
```
