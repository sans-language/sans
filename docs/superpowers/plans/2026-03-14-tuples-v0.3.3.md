# Tuples (v0.3.3) Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add tuple type to Sans — stack-allocated, fixed-size, heterogeneous value groups with positional access (`.0`, `.1`, `.2`).

**Architecture:** Tuples reuse the existing struct infrastructure (StructAlloc/FieldStore/FieldLoad IR instructions, LLVM struct codegen). No new IR instructions or codegen changes needed. The work is in parser, type checker, and IR lowering.

**Tech Stack:** Rust (compiler crates), Sans (test fixtures), LLVM (reuses existing struct codegen)

**Syntax (AI-optimized, no commas):**
```sans
t = (1 "hi" true)     // literal
t.0                    // access → 1
t.1                    // access → "hi"
f(a:I) (I S) = (a str(a))  // multi-return
```

```rust
TokenKind::LParen => {
    let start = tok.span.start;
    self.pos += 1;
    let first = self.parse_expr(0)?;
    if self.peek().kind == TokenKind::RParen {
        // Single expression in parens — grouping, not tuple
        self.pos += 1;
        Ok(first)
    } else {
        // Multiple expressions — tuple literal
        let mut elements = vec![first];
        while self.peek().kind != TokenKind::RParen && self.peek().kind != TokenKind::Eof {
            elements.push(self.parse_expr(0)?);
        }
        let rparen = self.expect(&TokenKind::RParen)?;
        Ok(Expr::TupleLiteral {
            elements,
            span: start..rparen.span.end,
        })
    }
}
```

```rust
fn parse_type_name(&mut self) -> Result<TypeName, ParseError> {
    // Tuple type: (I S B)
    if self.peek().kind == TokenKind::LParen {
        let start = self.peek().span.start;
        self.pos += 1; // consume (
        let mut inner_names = Vec::new();
        while self.peek().kind != TokenKind::RParen && self.peek().kind != TokenKind::Eof {
            let inner = self.parse_type_name()?;
            inner_names.push(inner.name);
        }
        let rparen = self.expect(&TokenKind::RParen)?;
        let name = format!("({})", inner_names.join(" "));
        return Ok(TypeName { name, span: start..rparen.span.end });
    }
```

```rust
Expr::TupleLiteral { elements, .. } => {
    let num_fields = elements.len();
    let dest = self.fresh_reg();
    self.instructions.push(Instruction::StructAlloc { dest: dest.clone(), num_fields });

    let ir_elements: Vec<IrType> = Vec::new();
    for (i, elem_expr) in elements.iter().enumerate() {

            field_index: i,
            value: val_reg,
```

## Implementation Notes

- **No new IR instructions** — tuples reuse StructAlloc, FieldStore, FieldLoad
- **No codegen changes** — tuples compile to LLVM structs identical to named structs
- **Parser disambiguation** — `(expr)` is grouping, `(expr expr ...)` is tuple (2+ elements)
- **Type representation** — tuple type `(I S B)` stored as string `"(I S B)"` in TypeName, resolved to `Type::Tuple { elements }` in typeck
- **Element access** — `.0`, `.1`, `.2` parsed as FieldAccess with numeric field name, dispatched in typeck and IR by checking for `Type::Tuple` / `IrType::Tuple`
